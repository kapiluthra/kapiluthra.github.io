Agentic ECM: When Your AI Decides Which Documents to Fetch

Single-shot retrieval can answer simple questions. But when a query requires fetching a contract, reading an annex, following an approval chain, and cross-referencing a policy — you need an agent, not a search engine.

Kapil Uthra · July 2026 · ~11 min read
Agentic AI ECM RAG OpenText LLM Tools Enterprise AI

The RAG Ceiling

Standard RAG — embed a query, fetch the top-k chunks, generate an answer — is a solved problem for simple factual lookups. "What is the refund policy?" works fine. But enterprise knowledge rarely packages itself into a single passage waiting to be retrieved.

Consider a more realistic query: "Has our master service agreement with Acme Corp been reviewed by Legal since the pricing amendment in Q3, and does it still comply with the new EU data residency requirements?"

That question requires an AI to: find the MSA, locate the Q3 amendment, check the Legal review workflow history, then read the EU compliance policy and reason across all of it. No vector search retrieves that in one shot.

This is the ceiling that agentic ECM is designed to break through. Instead of asking "what documents are most similar to this query?", an agent asks "what do I need to fetch next to make progress on answering this question?" — and keeps asking until it has enough.

ECM as a Tool Surface

The key insight behind agentic ECM is that an ECM system is already a structured tool surface — it just hasn't been exposed as one. OpenText Content Server offers everything an agent needs: full-text search, node fetch by ID, workspace traversal, metadata reads, version history, and workflow status. Wrap these as LLM-callable tools and the repository becomes programmable reasoning.

A minimal agentic ECM tool set looks like this:

ecm_tools.py — tool definitions
# Each tool maps to an OTCS REST endpoint wrapped with ACL enforcement

TOOLS = [
  {
    "name": "search_documents",
    "description": "Full-text search across the ECM repository. Returns node IDs, titles, and snippets.",
    "parameters": {"query": "string", "workspace_id": "int|null", "limit": "int=10"}
  },
  {
    "name": "fetch_document",
    "description": "Fetch the full text content of a document by its node ID.",
    "parameters": {"node_id": "int", "version": "int|null"}
  },
  {
    "name": "get_node_metadata",
    "description": "Read metadata for a node: owner, created date, categories, custom attributes.",
    "parameters": {"node_id": "int"}
  },
  {
    "name": "list_workspace_children",
    "description": "List immediate children of a folder or Business Workspace node.",
    "parameters": {"node_id": "int"}
  },
  {
    "name": "get_workflow_status",
    "description": "Check the approval/review workflow status for a document. Returns step, assignee, date.",
    "parameters": {"node_id": "int"}
  },
  {
    "name": "get_version_history",
    "description": "List versions of a document with timestamps and modifier names.",
    "parameters": {"node_id": "int"}
  }
]

Notice that workspace_id is an optional scoping parameter. Agents can start broad (search everywhere) or targeted (search within a specific project workspace). This turns out to be critical for performance — unconstrained searches over a large repository are slow, so good agent design encourages workspace-scoped tool calls after an initial locating step.

The Agent Loop

The agentic loop is straightforward to implement but surprisingly subtle to tune. At its core it is: plan → call → observe → decide → repeat until done.

Agentic ECM — reasoning loop
Input
User Query + Context
↓
LLM
Planner / Reasoner
↓ tool_call
ECM Tool
search / fetch / traverse
Guard
ACL Check + Hop Counter
↓ observation
Memory
Scratchpad / Context Window
↓ enough info? → answer : loop
Output
Final Answer + Source Citations
agent.py — core loop
import json
from dataclasses import dataclass, field
from typing import Any

@dataclass
class AgentState:
    query: str
    ticket: str                        # OTCS auth ticket (scoped to calling user)
    messages: list = field(default_factory=list)
    hop_count: int = 0
    MAX_HOPS: int = 10

def run_agent(query: str, ticket: str, llm, ecm_client) -> str:
    state = AgentState(query=query, ticket=ticket)
    state.messages.append({"role": "user", "content": query})

    while state.hop_count < state.MAX_HOPS:
        response = llm.chat(
            messages=state.messages,
            tools=TOOLS,
            tool_choice="auto"
        )

        if response.stop_reason == "end_turn":
            return response.content   # agent decided it has enough

        if response.stop_reason != "tool_use":
            raise RuntimeError(f"Unexpected stop: {response.stop_reason}")

        state.hop_count += 1
        tool_call = response.tool_use
        state.messages.append({"role": "assistant", "content": response.raw})

        # Execute the tool with ACL guard baked in
        result = execute_tool(tool_call, ticket=state.ticket, ecm=ecm_client)
        state.messages.append({
            "role": "tool",
            "tool_use_id": tool_call.id,
            "content": json.dumps(result)
        })

    return "[Agent exceeded maximum reasoning steps. Partial results may be incomplete.]"

Two things to notice. First, ticket is threaded through every tool call — there's no way for the LLM to bypass ACL checks by calling a tool directly. The execution layer always re-validates permissions using the authenticated user's ticket. Second, MAX_HOPS is a hard ceiling, not a soft suggestion. Without it, a confused agent can loop indefinitely burning both tokens and API quota.

Three Real ECM Agent Patterns

The tool surface above enables a whole category of agents that simply weren't possible with single-shot RAG. Here are three patterns I've implemented in production or close-to-production contexts.

Pattern 1: Contract Analysis Agent

Given a contract node ID and a compliance checklist, the agent: fetches the main contract, identifies amendment nodes from the version history, fetches each amendment, then reasons across the consolidated text to report which checklist items pass or fail — with citations to the specific clauses.

The trick here is that the agent must self-direct its traversal. It knows to look for amendments because the prompt gives it domain knowledge ("contracts may have associated amendment documents in the same workspace"). Without that domain hint in the system prompt, a general-purpose LLM will often just read the main document and stop.

contract_agent_system_prompt.txt
You are a contract analysis assistant with access to an enterprise document repository.

When analyzing a contract:
1. Always fetch the main contract document first.
2. List the workspace children to check for amendments, annexes, or addenda.
3. Fetch each amendment in date order and note any changes to key clauses.
4. Only then answer the user's question — citing specific sections by name.

Never assume a document is complete without checking for amendments.
Never cite text you have not fetched and read.
If a document is inaccessible (permission denied), note this explicitly.

Pattern 2: Onboarding Document Collector

Given a new employee's role and department, the agent: searches for the relevant onboarding workspace, traverses its folder structure, identifies which documents are tagged as "mandatory reading" vs optional, and returns a personalized reading list ordered by recommended sequence — without the HR team having to maintain a static list.

This pattern succeeds because ECM categories and custom attributes are machine-readable. The agent can filter by mandatory=true via get_node_metadata and sort by a reading_order attribute if it exists. The intelligence is in knowing to look for these attributes at all — again, a domain hint in the system prompt.

Pattern 3: Compliance Audit Agent

Given a regulation reference (e.g., "GDPR Article 30") and a set of business workspaces, the agent: searches for existing Records of Processing Activities, checks their last review dates, identifies which workspaces have no associated RoPA document, and produces a gap report. The loop can span 20–30 tool calls across dozens of workspaces.

⚠

Compliance audit agents must log every tool call and its result to an immutable audit trail — not just the final answer. Regulators want to see that the AI actually inspected the documents it claims to have reviewed.

The Hard Parts

Agentic ECM sounds powerful in the abstract. In practice, three problems dominate the engineering effort.

Tool latency compounds

A single fetch_document call on a large OTCS instance takes 200–800ms. At 10 tool calls per agent run, that's 2–8 seconds of wall-clock latency before the LLM even starts writing the answer. Users accustomed to sub-second RAG responses will find this jarring.

Mitigations: pre-fetch documents speculatively while the LLM is generating; cache heavily-accessed documents in a fast store with TTLs; encourage agents to batch related reads where possible. None of these eliminate the latency, but they shift it.

Hallucinated node IDs

LLMs have a tendency to invent plausible-looking node IDs when they can't find what they're looking for. A call to fetch_document(node_id=123456) on a non-existent node will return a 404 — but an unchecked agent will sometimes treat a 404 as a signal to try nearby IDs (123455, 123457) rather than reporting failure.

Fix: every tool must return a structured error envelope, and the system prompt must explicitly instruct the agent that a "not found" response means the document doesn't exist — not that the ID is slightly wrong.

ecm_client.py — structured error envelope
def fetch_document(self, node_id: int, ticket: str) -> dict:
    resp = self._get(f"/api/v1/nodes/{node_id}/content", ticket=ticket)

    if resp.status_code == 404:
        return {"ok": False, "error": "NOT_FOUND",
                "message": f"Node {node_id} does not exist. Do not guess adjacent IDs."}

    if resp.status_code == 403:
        return {"ok": False, "error": "PERMISSION_DENIED",
                "message": f"Node {node_id} exists but you lack See Contents permission."}

    return {"ok": True, "node_id": node_id, "content": resp.text[:MAX_CONTENT_CHARS]}

Infinite reasoning loops

The subtlest failure mode is an agent that is genuinely uncertain but keeps searching rather than admitting it. It fetches document A, which mentions document B. It fetches B, which references policy C. Policy C refers to a guideline that expired in 2021. The agent searches for the updated guideline, can't find it, and starts searching more broadly. Meanwhile the hop counter is climbing toward MAX_HOPS.

The solution is a progress signal: after every tool call, the agent should append a one-line scratchpad note summarising what it just learned and whether it is closer to an answer. When the scratchpad shows no new information for two consecutive steps, the agent should be instructed to conclude with what it has.

system_prompt_anti_loop.txt
After each tool call, before calling the next tool, write:
PROGRESS: [one sentence describing what you just learned]
NEXT: [one sentence describing what you still need to find, or "READY TO ANSWER"]

If NEXT is the same as the previous step's NEXT, you are looping.
Stop, summarise what you found, and state clearly what you could not locate.

Guardrails in Production

Beyond MAX_HOPS, a production-grade agentic ECM system needs three additional guardrails to be deployable in an enterprise context.

ACL checks on every fetch

This one cannot be stressed enough. The agent's ticket is the user's ticket — not an admin service account. Every tool call must pass the ticket through to OTCS, which enforces ACLs natively. Agents that use a shared service account for retrieval will leak documents across permission boundaries. The ACL-aware architecture described in ACL-Aware RAG applies here verbatim: the only safe place to enforce permissions is at fetch time, in the tool implementation, using the authenticated user's credentials.

Immutable audit logging

Every tool call and its result must be written to an append-only log: who called the agent, when, with what query, which nodes were fetched, and what answer was returned. This is non-negotiable in regulated industries. The log also becomes invaluable for debugging when an agent produces a wrong answer — you can replay every step it took.

audit_logger.py
import time, json, hashlib

class AuditLogger:
    def __init__(self, session_id: str, user_id: str, store):
        self.session_id = session_id
        self.user_id = user_id
        self.store = store           # write-once object store or append-only DB
        self.entries = []

    def log_tool_call(self, tool: str, params: dict, result: dict):
        entry = {
            "ts": time.time(),
            "session": self.session_id,
            "user": self.user_id,
            "tool": tool,
            "params": params,
            "result_ok": result.get("ok"),
            "result_hash": hashlib.sha256(json.dumps(result).encode()).hexdigest()
        }
        self.entries.append(entry)
        self.store.append(entry)     # write immediately, not at session end

    def log_answer(self, answer: str, hop_count: int):
        self.store.append({
            "ts": time.time(), "session": self.session_id,
            "type": "answer", "hops": hop_count,
            "answer_hash": hashlib.sha256(answer.encode()).hexdigest()
        })

Content size limits

Enterprise documents can be large — a 200-page contract will overflow any LLM context window. The tool must truncate at a safe character limit and signal that truncation occurred. Agents should be instructed to fetch specific sections (by heading search) rather than entire documents when they already know what they're looking for.

✓

For large documents, expose a fetch_section(node_id, heading) tool that returns only the text under a specific heading. Agents learn to use it after being burned by truncated fetches on full-document calls.

Where Agentic ECM Stands Today

The technology is real and the patterns work — I have running implementations of all three agent patterns described above. But "works in a controlled environment" is a long way from "enterprise production-ready." The gap is not in the AI capabilities; it's in the surrounding infrastructure.

What's missing at most organisations today:

  • No structured error envelopes from ECM APIs (4xx responses return HTML error pages, not JSON)
  • No per-user audit log infrastructure (most ECM audit logs are admin-only and not queryable by AI systems)
  • No reliable ACL APIs for external callers (ticket-based auth works but isn't designed for high-concurrency agent workloads)
  • No content-size metadata before fetch (agents can't know a document is 500 pages until they try to fetch it)
  • No "intent-aware" ECM search (full-text search returns keyword matches; agents need semantic relevance ranking)

Each of these is solvable. Structured error envelopes are a thin wrapper layer. ACL concurrency limits can be managed with a connection pool. Content-size metadata is usually available via node metadata before content fetch. Semantic search is the RAG infrastructure problem already solved in earlier articles.

The honest assessment: agentic ECM is 18–24 months away from mainstream enterprise deployment at most large organisations, not because the AI isn't ready, but because the ECM infrastructure wasn't designed for agents as clients. The organisations that start instrumenting their ECM APIs today — adding structured errors, per-user audit trails, semantic search, and content-size metadata — will be ready to deploy agents when the LLMs and tooling mature. Everyone else will spend that time on infrastructure catch-up.
