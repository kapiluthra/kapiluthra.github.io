Clean Architecture in Enterprise ECM: What Actually Works

13 years of ECM projects have given me opinions. On abstraction layers, on LlamaIndex, on vendor lock-in, and on when to build versus buy. These are the ones I'd argue for at 11pm before a go-live.

Kapil Uthra March 2026 ~10 min read
architectureopinionecmrag

I want to write about architecture principles that I've actually found valuable — not the kind that sound good in a slide deck but evaporate on contact with a real enterprise deployment. These are opinions I've formed by being wrong first, then figuring out why.

The Abstraction Layer Problem

The most common architectural mistake I see in ECM integrations is insufficient abstraction over the document repository. Code reaches directly into OpenText API calls, Documentum API calls, or SharePoint REST calls scattered throughout the application. Then the client says they want to migrate to a different system in three years, and you discover there's no seam to cut at.

The pattern I use now: define your own repository interface — a small set of operations (fetch_document, list_children, get_metadata, download_content) — and write an OTCS adapter that implements it. Your application code only talks to the interface. When you need to switch to a different system, you write a new adapter. This sounds obvious but I've seen it ignored on projects with 100-person teams.

For my RAG pipeline, the interface boundary is explicit: the CSClient is not imported anywhere in the chunker, embedder, or retriever. Only the ingestion orchestrator touches it. This meant when I needed to test the chunking and embedding logic, I could do so with local files without needing a live OTCS connection.

On LlamaIndex and Frameworks

LlamaIndex and LangChain both have OTCS connectors. I've evaluated both, and my position is: use them for prototyping, understand what they're doing, then consider whether you need them in production.

The frameworks abstract away the chunking, embedding, and retrieval loops in ways that are genuinely convenient. They also make certain assumptions about your data pipeline that are fine for general use but wrong for enterprise ECM — particularly around ACL enforcement, incremental sync, and the cold/hot store separation I described in the RAG pipeline post. When a framework assumption conflicts with an enterprise requirement, you end up fighting the framework.

The question isn't "should I use a framework?" but "do I understand this system well enough that the framework is helping me rather than hiding complexity I need to see?"

For a 500-document RAG demo, use LlamaIndex. For a production system with ACL gates, incremental sync, and enterprise SLAs, know the framework deeply enough to know where to deviate from it.

Vendor Lock-in in ECM Is Real, and the Answer Isn't Avoidance

Every meaningful ECM deployment is vendor-locked. The metadata model is vendor-specific. The permission model is vendor-specific. The workflow engine is vendor-specific. You can't meaningfully abstract these away — if you try, you end up with either a lowest-common-denominator system that uses none of the platform's actual capabilities, or so many leaky abstractions that you have both the complexity of the abstraction and the complexity of the underlying system.

The honest architectural answer to vendor lock-in is: make the lock-in explicit and bounded. Decide which systems are the platform foundation (OTCS, Oracle DB, OTDS — these you're committed to), and build everything else in a way that doesn't assume them. Your business logic, your workflow rules, your reporting — these should be testable without the vendor platform. Your repository adapter is explicitly coupled to the platform and that's fine.

Build vs Buy: The Real Question

I've been asked "should we build a custom migration tool or use a third-party?" and "should we build a RAG chatbot or buy OpenText's AI product?" more times than I can count. The framework I use:

  • Buy when the problem is not differentiated. Migrations, ETL, monitoring, backup. The third-party tool is almost always better than what you'll build, the vendor is invested in keeping it working with new versions, and your engineers' time is worth more than the license cost.
  • Build when the problem is core to your competitive advantage, when your specific requirements can't be met by available tools, or when the total cost of adapting a tool to your needs exceeds the cost of building.
  • Start with buy, migrate to build when you need to ship something working now but have specific requirements that the off-the-shelf solution won't satisfy long-term. Use the commercial tool to learn the problem space, then build the right solution.

For the RAG pipeline: I looked at OpenText's native AI capabilities, LlamaIndex's OTCS connector, and building from scratch. I built from scratch — not because the tools were bad, but because the specific requirements (ACL gates, parent-child chunking, hybrid RRF reranking, cold store for re-embedding) were specific enough that adapting the frameworks would have taken as long as building and left me with less control. For a simpler use case with standard requirements, I'd use the framework.

The Infrastructure is the Product

One opinion I hold strongly: in enterprise ECM, the infrastructure decisions are as important as the application decisions, and they deserve the same architectural attention. How the database is sized and maintained, what the backup/recovery strategy is, how the OTCS nodes scale under load — these aren't operational afterthoughts, they're part of the system design.

The projects that struggle in year three are almost always ones where the infrastructure was stood up quickly to get the application running, then never revisited. Index fragmentation accumulates. Version tables explode. The recovery procedure was never tested. Nobody knows who owns the OTDS configuration.

Architectural rigor applied to infrastructure looks like: documented sizing decisions and their assumptions, a tested backup/recovery procedure, a database maintenance schedule, and explicit ownership of every component. This is boring work. It's the kind of work that determines whether your ECM deployment is healthy at year five.

The most reliable ECM architectures I've seen are the most boring ones: standard configurations, conservative customization, clear ownership, documented runbooks. The exciting, innovative ones that push the platform boundaries are the ones I get called in to stabilize.
