Keeping Your OTCS RAG Pipeline Fresh: Incremental Sync with OTModifyDate

A full crawl of 500+ documents takes time. Running it daily to catch updates is wasteful. Here's the pattern I use to detect and re-ingest only changed documents — using the field the OTCS API already exposes.

Kapil Uthra April 2026 ~10 min read
rag opentext incremental sync python chromadb

The first version of most RAG pipelines is a batch job: crawl everything, embed everything, store everything. It works fine for demos. In production, you hit the problem that makes it unsustainable: your knowledge base goes stale the moment you finish building it.

In an enterprise ECM environment, documents are updated constantly — contracts get amended, policies get revised, project documents accumulate new versions. A RAG system that doesn't track these changes starts giving confidently wrong answers, which is worse than giving no answer at all.

The fix is incremental sync. And OpenText Content Server gives you exactly the field you need to implement it: modify_date.

The Core Idea

Every node in Content Server exposes a modify_date timestamp — updated whenever the document content or metadata changes. The incremental sync strategy is simple:

  1. Record the timestamp when your last successful sync completed
  2. On the next sync run, walk the node tree as normal
  3. For each document, compare its modify_date to your last sync timestamp
  4. Only re-process documents where modify_date > last_sync
  5. For changed documents: delete existing chunks, re-extract, re-chunk, re-ingest
  6. Update the sync timestamp to now

This means each sync run processes only the documents that actually changed. On a corpus of 500 documents where 10–20 change per day, you go from processing 500 documents per run to processing 10–20. The time saving compounds quickly.

ℹ

Important: The modify_date field reflects the last modification to either the document content or its metadata (categories, permissions, name). A permission change on a document will trigger re-ingestion — which is correct behaviour if you're enforcing ACL gates in your retriever.

Sync State Management

The sync state needs to survive between runs. I use a simple JSON file — one field, the ISO 8601 timestamp of the last successful sync. For production deployments you'd move this to a database, but the JSON approach is fine for single-process pipelines.

sync_state.py — Persistent sync timestamp
import json
import pathlib
from datetime import datetime, timezone

SYNC_STATE_FILE = pathlib.Path("sync_state.json")

def get_last_sync() -> str:
    """Return ISO 8601 timestamp of last successful sync.
    
    Returns epoch start if no sync has been run yet — triggers a full
    initial crawl on first run.
    """
    if SYNC_STATE_FILE.exists():
        data = json.loads(SYNC_STATE_FILE.read_text())
        return data.get("last_sync", "2000-01-01T00:00:00Z")
    return "2000-01-01T00:00:00Z"  # first run = full crawl

def save_sync_time(timestamp: str = None) -> None:
    """Persist sync completion time. Call ONLY after successful sync."""
    ts = timestamp or datetime.now(timezone.utc).isoformat()
    SYNC_STATE_FILE.write_text(json.dumps({
        "last_sync": ts,
        "last_sync_human": ts  # readable copy for debugging
    }, indent=2))

def needs_update(node: dict, last_sync: str) -> bool:
    """Return True if this node was modified after last_sync."""
    modify_date = (
        node.get("data", {})
            .get("properties", {})
            .get("modify_date", "")
    )
    if not modify_date:
        return True  # no date = assume changed (safe default)
    return modify_date > last_sync

The string comparison in needs_update works because OTCS returns modify_date in ISO 8601 format (2026-04-10T14:23:01Z), which sorts correctly as a string. No datetime parsing needed.

The Incremental Sync Loop

sync.py — Main incremental sync orchestrator
import logging
from cs_client import CSClient
from extractor import extract_text
from chunker import chunk_document
from ingester import ingest_chunks, delete_doc_chunks
from sync_state import get_last_sync, save_sync_time, needs_update

logger = logging.getLogger(__name__)

def run_sync(client: CSClient, root_node_id: int) -> dict:
    """Run an incremental sync of the OTCS repository.
    
    Returns stats dict with counts of processed/skipped/failed documents.
    """
    last_sync = get_last_sync()
    logger.info("Starting sync. Last sync: %s", last_sync)

    stats = {"processed": 0, "skipped": 0, "failed": 0, "deleted": 0}

    for node in client.walk_nodes(root_node_id):
        props = node["data"]["properties"]
        doc_id = str(props["id"])
        doc_name = props.get("name", "unknown")

        if not needs_update(node, last_sync):
            stats["skipped"] += 1
            continue

        logger.info("Processing: %s (%s)", doc_name, doc_id)

        try:
            # 1. Download content
            content, mime_type = client.download_content(int(doc_id))
            if not content:
                stats["skipped"] += 1
                continue

            # 2. Extract text
            text = extract_text(content, mime_type)
            if not text.strip():
                stats["skipped"] += 1
                continue

            # 3. Delete existing chunks (upsert semantics)
            delete_doc_chunks(doc_id)
            stats["deleted"] += 1

            # 4. Re-chunk and re-ingest
            chunks = chunk_document(
                text=text,
                doc_id=doc_id,
                doc_name=doc_name,
                modify_date=props.get("modify_date", ""),
                mime_type=mime_type,
            )
            ingest_chunks(chunks)
            stats["processed"] += 1

        except Exception as e:
            logger.error("Failed: %s (%s): %s", doc_name, doc_id, e)
            stats["failed"] += 1

    # Save sync time ONLY if no failures — safe-fail approach
    if stats["failed"] == 0:
        save_sync_time()
        logger.info("Sync complete: %s", stats)
    else:
        logger.warning(
            "Sync had %d failures — not advancing sync timestamp. "
            "Failed docs will be retried next run.",
            stats["failed"]
        )

    return stats
⚠

Critical detail: only advance the sync timestamp if the run completed without failures. If you save the timestamp after a partial run, the failed documents will never be retried — they fall before the new timestamp and the next sync skips them. Failed documents must stay behind the watermark.

Handling Deletions

The sync loop above handles new and modified documents. But what about deleted documents? If a document is deleted from OTCS, walk_nodes won't yield it — so its stale chunks stay in ChromaDB forever, available to the retriever.

The clean solution is a periodic reconciliation pass — run less frequently than the incremental sync (weekly rather than daily):

reconcile.py — Remove stale chunks for deleted documents
from ingester import _get_collection

def reconcile_deletions(client: CSClient, root_node_id: int) -> int:
    """Remove ChromaDB chunks for documents that no longer exist in OTCS."""
    # Build set of current OTCS doc IDs
    live_ids = {
        str(client.get_properties(node)["id"])
        for node in client.walk_nodes(root_node_id)
    }

    # Get all doc IDs in ChromaDB
    collection = _get_collection()
    all_meta = collection.get(include=["metadatas"])
    indexed_ids = {m["doc_id"] for m in all_meta["metadatas"]}

    # Delete chunks for docs that are in ChromaDB but not in OTCS
    stale_ids = indexed_ids - live_ids
    deleted_count = 0
    for doc_id in stale_ids:
        delete_doc_chunks(doc_id)
        deleted_count += 1

    return deleted_count

Why the Parquet Cold Store Pays Off Here

When you delete and re-ingest a modified document, you're re-embedding it — which costs compute time. If you later decide to switch embedding models (say, from nomic-embed-text to a newer model), you'd need to re-embed everything.

Without the Parquet cold store, that means re-crawling all of OTCS — all the API calls, all the downloads, all the text extraction. With the cold store, re-embedding is just reading the Parquet file locally and sending text to the embedding API. The cold store pays for itself the first time you change models.

✓

Re-embedding from cold store: read all rows from Parquet, call the new embedding model on each child_text, upsert into ChromaDB with the new vectors. No OTCS API calls required. For 500 documents this takes minutes rather than hours.

Scheduling the Sync

The incremental sync is lightweight enough to run frequently. The pattern I use:

Sync Schedule
1×
Initial full crawl
Run once to bootstrap the knowledge base. Takes longest — all documents processed.
4× daily
Incremental sync
Runs every 6 hours via cron or Kubernetes CronJob. Only processes documents modified since last run — typically 5–30 documents on an active repository.
1× weekly
Deletion reconciliation
Full walk to find and remove chunks for deleted documents. Keeps ChromaDB consistent with the live repository.
As needed
Model migration
When switching embedding models: re-embed all docs from Parquet cold store. No OTCS API calls. Typically 15–45 minutes for 500 documents.

Edge Cases Worth Knowing

  • New documents: handled automatically. A new document has a modify_date after your last sync timestamp, so it gets processed on the next run.
  • Version-only updates: Content Server updates modify_date when a new version is added. This correctly triggers re-ingestion of the new version's content.
  • Metadata-only changes: renaming a document or changing its category also updates modify_date. The document gets re-ingested with updated metadata — correct if your chunks store document name and category.
  • Permission changes: if you're using ACL security gates in your retriever (you should be), permission changes also correctly trigger re-ingestion, ensuring the metadata the retriever uses for access checks stays current.
  • Clock skew: if your OTCS server and your sync process are on different machines with unsynchronised clocks, you can get missed updates. Add a small overlap window — save last_sync as 5 minutes before the actual sync start time to catch anything that falls in the gap.
sync_state.py — Clock skew buffer
from datetime import datetime, timezone, timedelta

def save_sync_time_with_buffer(buffer_minutes: int = 5) -> None:
    """Save sync time with a small lookback buffer for clock skew safety."""
    buffered = datetime.now(timezone.utc) - timedelta(minutes=buffer_minutes)
    SYNC_STATE_FILE.write_text(json.dumps({
        "last_sync": buffered.isoformat()
    }, indent=2))

The 5-minute buffer means a small number of unchanged documents get re-checked on each run (those modified in the last 5 minutes of the previous window), but you'll never miss an update due to clock skew. On a 500-document corpus, the false-positive rate is negligible.

This whole sync pattern — modify_date watermark, safe-fail timestamp advancement, cold store re-embedding, deletion reconciliation — is what separates a RAG demo from a RAG system you can actually depend on in production. The code is in the demo repo if you want to start from a working scaffold.
