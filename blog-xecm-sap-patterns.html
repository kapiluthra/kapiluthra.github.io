xECM for SAP: Architecture Patterns That Actually Work

After implementing Extended ECM for SAP across 10+ engagements — pharma, manufacturing, chemicals, utilities — here are the patterns that hold up under production load, and the anti-patterns I keep seeing teams repeat.

Kapil Uthra March 2026 ~14 min read
xecmsapopentextarchitectureenterprise

Extended ECM for SAP (xECM) is deceptively complex. On paper, it's a connector between SAP and OpenText Content Server — documents live in OTCS, SAP shows them in context. In practice, you're aligning two very different data models, two authentication systems, and two sets of governance rules, while keeping both systems performant under enterprise load.

I've been the lead architect on xECM for SAP implementations for companies across five countries, covering MM/PM/HR modules, GxP-regulated pharma environments, and migrations from legacy DMS systems. Here's what I've learned works — and what doesn't.

Business Object Architecture

The foundation of every xECM deployment is the business object model: which SAP objects get their own OTCS workspace, what folders and categories live inside those workspaces, and how the workspace creation is triggered.

The most common mistake I see is over-granular business object definition. Teams try to create workspaces for every SAP object type — Purchase Order, Purchase Requisition, Goods Receipt, Invoice, Line Item. The result is a Content Server with millions of near-empty workspaces, terrible performance on workspace lookup, and a maintenance nightmare.

❌ Anti-Pattern

Workspace per transaction: PO + PR + GR + IR = 4 workspaces per procurement cycle. Millions of workspaces, most with 1–2 documents.

✓ Pattern

Workspace at the business process level: one workspace per Vendor + Procurement cycle. Child folders for transaction types. Volume scales linearly with business, not transactions.

For a manufacturing client managing 20K workspace creations per day (TATA Steel), the difference between these approaches is the difference between a sustainable system and a system that degrades linearly with usage.

Workspace Creation: Synchronous vs Asynchronous

When does the OTCS workspace get created? There are two models:

  • Synchronous: workspace created at SAP object creation time (PO save triggers workspace creation). Simple, but adds latency to SAP transactions and creates coupling.
  • Asynchronous: SAP object created normally, workspace created later by a batch job or event queue. More complex, but SAP performance is unaffected and failures are isolated.

For high-volume environments (500+ workspaces/day), asynchronous creation is the right choice. The batch job runs every 15–30 minutes, picks up new SAP objects via BAPI/RFC, and creates workspaces in OTCS. Any failures are retried without impacting SAP operations.

⚠

GxP environments: Synchronous creation is often required for audit trail completeness — regulators want proof that the workspace existed before any document was generated. At a Finnish pharmaceutical client, we implemented synchronous creation with a 2-second timeout and graceful degradation: SAP object saves even if OTCS workspace creation fails, with a reconciliation job to create missing workspaces.

OTDS: The Authentication Bridge

OpenText Directory Services (OTDS) is the SSO layer between SAP and Content Server. Getting this right is critical — misconfigured OTDS is the most common source of "user can log into SAP but gets 401 in the document panel" support tickets.

The three things that always need to match exactly: user partition names, login attribute mappings (SAP user ID → OTDS principal), and resource partition assignment. A mismatch in any of these produces an error that looks like a permissions problem but is actually an identity federation failure.

I always recommend running OTDS in a dedicated VM or container separate from the Content Server host. OTDS has different memory and CPU characteristics than OTCS, and co-locating them causes contention that manifests as intermittent authentication failures under load — the hardest class of issue to diagnose.

Database Maintenance Strategy

At 10K+ active users and years of accumulated documents, Content Server database performance is the dominant operational concern. The patterns that matter:

  • DTree and DAuditNew tables grow unbounded. DTree holds all node records; DAuditNew holds audit events. Both need periodic archiving or you'll see query plans degrade as the tables hit millions of rows.
  • Index rebuild scheduling should happen weekly during off-peak windows. OTCS ships with admin console scripts for this. Skipping it for more than a month will cause noticeable slowdowns on search and folder expand.
  • Version pruning: by default, OTCS keeps unlimited document versions. Set a version retention policy early. We defined 5 major versions + 30 days of minors for most content types. Without this, version tables explode.
  • Statistics updates: Oracle and SQL Server both need regular statistics updates on the OTCS schema. This is separate from index rebuilds and equally important.

The Workspace Template Problem

xECM workspaces are created from templates defined in Content Server. The classic mistake is creating workspace templates in the live repository rather than in a dedicated template area. When the template structure needs to change (and it will — business requirements evolve), you can update the template and new workspaces get the new structure — but existing workspaces don't. Now you have two generations of workspace structure with no migration path.

The pattern I use: version your workspace templates explicitly. When a major structural change is needed, create a new template version. Write a migration script that updates existing workspaces to the new structure. Test it on 100 workspaces, then on 1000, then run it production. This is more upfront work but avoids the structural drift that makes 5-year-old xECM deployments unmaintainable.

Performance: The 19 Business Objects Case

At TATA Steel, we implemented 19 business objects across MM, PM, and HR modules for 10K active users. A few performance decisions that made a material difference:

  • Physical separation of high-volume and low-volume workspaces. MM Purchase Order workspaces (millions) live in a different Content Server partition from HR Personal File workspaces (tens of thousands). This prevents high-volume MM operations from competing with HR document access.
  • Faceted search over full-text search for workspace lookup. SAP users finding a PO workspace by business ID should use a metadata search against the OTCS category, not a full-text index scan. Much faster, deterministic results.
  • Pre-creating folder structure on workspace creation. Creating the folder hierarchy lazily (on first document upload) causes spiky performance. Pre-create the full structure at workspace creation time, even if most folders stay empty.

What I'd Do Differently

Looking back across all the xECM SAP projects: the technical decisions rarely cause failures. What causes failures is scope creep in business object definition, under-investment in OTDS setup, and skipping the database maintenance planning until performance problems appear in production.

If I were starting a new engagement today, the first three days would be: (1) define and lock down the business object inventory before any technical setup, (2) set up OTDS in a dedicated environment and validate SSO end-to-end before any other configuration, (3) agree on a database maintenance schedule and version retention policy in writing. Everything else is tractable.
