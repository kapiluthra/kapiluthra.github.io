Migrating 150M Documents to Kubernetes-Native Content Server

Architecture, tooling, and hard lessons from the Securex Belgium engagement — moving 150 million documents from a legacy ECM system into a cloud-native OpenText Content Server deployment, with zero data loss.

When Securex Belgium approached us at Turnpikes, they had a specific problem: 150 million documents sitting in a legacy ECM system that had reached end-of-life, and a hard deadline to complete the migration. The target was a Kubernetes-native OpenText Content Server deployment, also integrated with their Salesforce CRM through Extended ECM for Salesforce.

This is the architecture writeup. What we built, how the migration tooling worked, and what surprised us along the way.

Why Kubernetes for Content Server

Content Server on Kubernetes is fundamentally different from traditional on-premise or even cloud-VM deployments. The main benefits are horizontal pod autoscaling (you can spin up additional CS nodes during migration bursts and scale back down), rolling deployments with zero downtime, and native integration with cloud storage backends (S3 or Azure Blob) for the binary object store.

The main challenge is stateful storage. Content Server has two storage concerns: the database (Oracle or SQL Server — must run outside the cluster on a managed service) and the OTCS file store (the binary objects — which maps to S3 or a persistent volume). Getting the persistent volume and storage class configuration right is essential. We learned this the hard way on an earlier engagement where the wrong storage class caused I/O latency that looked like Content Server performance issues.

✓

Storage class matters: For the OTCS file store on AKS or EKS, use premium SSD storage classes for the primary node and blob/object storage for archive tiers. The default standard storage classes that come with managed Kubernetes will cause performance issues at scale.

The Migration Architecture

We used the Turnpikes Migration Tool — a tool I product-managed — as the core migration engine. The overall flow breaks into three phases: extraction (read from source system), transformation (map metadata, clean data, build OTCS target model), and load (write into Content Server via API or direct DB + file store injection).

For 150M documents, API-only ingestion is too slow. We used a hybrid approach: metadata goes through the OTCS REST API (which handles the DTree entries, category assignments, and permission sets), while binary content is injected directly into the S3 file store with checksums verified post-injection. This reduced ingestion time from an estimated 180 days (API-only) to under 6 weeks.

Metadata Mapping: The Real Work

Binary injection is the fast part. The slow, difficult, high-risk part is metadata mapping. The source system used a flat metadata model with 40+ fields, some populated inconsistently over 15 years of data entry. The target Content Server model used OTCS Categories with strict attribute types.

The transformation layer had to handle:

  • Date normalization: source data had at least 6 different date formats. We wrote an explicit parser chain and logged every document where date parsing required fallback logic.
  • Owner mapping: source user IDs from the legacy system needed to map to OTCS user accounts. Some legacy users no longer existed; we created a "migrated content" system account as the default owner with the original owner stored in a custom attribute.
  • Category assignment: not all documents could be auto-categorized. The transformation layer classified documents using filename patterns and source-system metadata, then flagged uncertain classifications for manual review. We ended up manually reviewing ~2% of documents.
  • Permission inheritance: the legacy system had custom permission sets that didn't map cleanly to OTCS ACLs. We defined a mapping table and validated it with the client's security team before the migration run.

Parallel Migration Streams

150M documents can't run in a single sequential process. We partitioned the source corpus into 20 streams by document category and ran them in parallel, each targeting a separate Kubernetes Job. This gave us:

  • Fault isolation: a failure in one stream doesn't affect others. We could resume failed streams from their last checkpoint.
  • Progress visibility: each stream reported its count and rate independently. We could see exactly which categories were slow and why.
  • Load distribution: streams targeting high-volume document types (invoices, contracts) were allocated more Kubernetes resources than low-volume types.

Checkpointing was implemented at the document level using a PostgreSQL tracking table. Every document got a status: pending, extracted, transformed, loaded, verified, or failed. If a stream crashed, it resumed from the last verified checkpoint.

Verification: Trust Nothing Until Proven

For a migration of this scale, you need systematic verification — not spot checks. Our verification pipeline ran after every batch and confirmed:

  • Row count parity: documents in source vs documents in OTCS target, by category.
  • Checksum verification: SHA-256 of every binary object in source compared to OTCS file store. This caught two edge cases where the S3 injection had silently corrupted files (the culprit was a character encoding issue in filenames with special characters).
  • Metadata spot checks: statistical sample of 1000 random documents comparing source metadata to OTCS Category attributes.
  • Permission audit: for a random sample of 500 documents, verify that the migrated OTCS permissions match the source system permissions.

The checksum verification step caught 147 corrupted documents out of 150 million — a rate of about 0.0001%. Trivial in percentage terms, but 147 lost documents would have been unacceptable to the client. We re-migrated all 147 successfully.

The Salesforce Integration Layer

Alongside the migration, we implemented Extended ECM for Salesforce — so sales users could access relevant documents directly from their Salesforce CRM records. This required mapping Salesforce Account and Opportunity IDs to OTCS workspace structures created during migration.

⚠

Timing matters: the xECM for Salesforce configuration must be in place before migration completes — not after. Workspaces need to be created with the correct Business Workspace type for Salesforce integration to link correctly. Retrofitting the workspace type on millions of nodes after migration is not fun.

What I'd Do Differently

The migration completed successfully — 150M documents, zero data loss, on schedule. But there were things I'd approach differently next time:

  • Start checkpointing earlier in design. We added document-level checkpointing partway through development. Starting with it would have saved two weeks of rework.
  • Run a 1M document dry-run on production infrastructure before committing to the full migration. We ran our dry-runs on staging, which had different storage performance characteristics than production. The production run initially ran slower than projected until we tuned the S3 injection batch sizes.
  • Plan the post-migration indexing time explicitly. After loading 150M documents, Content Server's full-text search index needed significant time to catch up. Users expected search to work immediately after go-live. We should have communicated the indexing window more clearly in advance.

Large-scale ECM migrations are one of the more complex, less glamorous problems in enterprise IT — there's no clever algorithm that makes 150 million files move faster. What matters is systematic engineering: good tooling, careful metadata work, verification at every stage, and the discipline to not skip steps when schedule pressure builds.
