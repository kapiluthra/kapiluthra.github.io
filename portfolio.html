Securex · Belgium · 2024
150M Document Migration into Kubernetes-Native Content Server + xECM for Salesforce
Lead Architect Migration · Cloud · Integration ✓ Zero data loss · On schedule

The Challenge
Securex Belgium needed to exit a legacy ECM at end-of-life, landing 150 million documents in a Kubernetes-native OTCS deployment — simultaneously integrating with Salesforce CRM via xECM. Hard deadline, zero data loss tolerance, flat source metadata that needed mapping to a structured OTCS Category model.
Architecture Decisions
Hybrid injection: metadata via OTCS REST API; binary content injected directly to S3 with post-injection SHA-256 verification. This cut migration time from ~180 days (API-only) to 6 weeks. 20 parallel Kubernetes Jobs by document category, each with per-document PostgreSQL checkpointing. xECM for Salesforce workspace type configured before migration — critical for correct linkage.
What Surprised Us
SHA-256 verification caught 147 corrupted files out of 150M (0.0001%) — a character encoding edge case in filenames with special characters. All 147 re-migrated successfully. Without checksum verification this would have been silent data corruption.
What I'd Do Differently
Start with per-document checkpointing in design (we added it mid-development). Run the 1M-doc dry run on production infrastructure, not staging — storage I/O differences caused initial rate underestimates. Communicate the post-migration full-text indexing window to users on day one.

The Challenge
Orion is a Finnish pharma manufacturer under EU GxP / 21 CFR Part 11 requirements. Their document management system is part of the validated environment — every configuration change requires documented evidence. Standard xECM implementation timelines don't account for validation overhead.
GxP Overhead (What Others Skip)
Change Control submitted 6 weeks pre-implementation. IQ/OQ/PQ protocols written with full traceability matrix. Post-go-live audit trail captured in OTCS audit log itself — used as primary IQ evidence. This overhead roughly doubled project timeline vs non-regulated xECM. Worth every day.
Key Lesson
In GxP environments, the ECM system is part of the product quality infrastructure, not an IT tool. Treat validation as a first-class deliverable from day one — retrofitting evidence after implementation is significantly more expensive.
OTDS Config Note
OTDS was configured on a dedicated VM (not co-located with OTCS) — separate memory/CPU profiles prevent the contention that causes intermittent auth failures under load. This became a standard practice across all engagements after this project.

SCG · USA · 2022–2023
30-Server AWS Landscape — 90% Upgrade Outage Reduction via Rolling Deployment
Lead Architect AWS · Distributed ECM · DR ✓ 8–12 hr outage window → 45 min

The Problem
SCG's 30+ server Content Server landscape on AWS required 8–12 hour outage windows for major version upgrades. Significant business disruption and compliance risk around change management. All nodes had to stop simultaneously because session state was node-local.
The Solution
Moved all CS session state to an external Redis cluster — making any node able to serve any session regardless of version. This enabled true rolling upgrades: each front-end node is pulled from the ALB, upgraded, verified, and returned, while users continue on remaining nodes. Only the database schema step (unavoidable) requires a brief full outage.
DR Architecture
Alongside the upgrade redesign: cross-AZ standby, automated ALB failover via Route53 health checks, documented RTO/RPO targets. The previous single-region setup couldn't meet SCG's business continuity requirements — this engagement addressed both upgrade pain and DR gap in a single architecture revision.
Transferable Pattern
Session externalisation is now a standard recommendation for any multi-node OTCS deployment — it improves HA beyond just enabling rolling upgrades. Any node failure becomes transparent to users rather than requiring a session re-login.

JIO · India · 2014–2015

The Challenge
JIO India's ECM needed to sustain 2 million document ingestions per day from billing, contract, and customer management workflows. The existing Windows-based deployment was degrading under this load. A 65 TB archive migration and platform transition (Windows → Linux) were required simultaneously.
Performance Remediation
Root causes: severe DTree/DAuditNew index fragmentation from years of deferred maintenance, undersized DB connection pool for the ingestion concurrency, and I/O contention between the full-text indexer and document ingestion on the same storage. Resolved by separating the indexer to a dedicated node and implementing a proper maintenance schedule.
