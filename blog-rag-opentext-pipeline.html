Building a Production RAG Pipeline for OpenText Content Server

End-to-end walkthrough: OTCSTicket auth, document discovery, chunking strategies, ChromaDB ingestion, and hybrid retrieval with RRF reranking — from a 14-year ECM veteran who actually built this.

Kapil Uthra April 2026 ~18 min read
rag opentext chromadb python ollama enterprise

Most RAG tutorials start with a folder of PDFs and a simple vector store. That's fine for demos. But if you're working in an enterprise environment with OpenText Content Server as your document repository, you face a completely different set of problems: authentication tokens that expire mid-crawl, deeply nested folder hierarchies, container nodes vs document nodes, ACL-based access control, and a REST API that has a few non-obvious quirks if you don't know where to look.

I've spent 14 years working with OpenText Content Server — from version 9.7 through current cloud-native deployments. Recently, I built a production RAG pipeline targeting 500+ enterprise documents in OTCS. This article is the writeup I wish had existed when I started.

ℹ

The stack: Python 3.11, ChromaDB (vector store), Ollama with nomic-embed-text embeddings, Streamlit (chatbot UI). OTCS version 23.4 on Kubernetes. The patterns apply broadly to any recent OTCS version.

Architecture Overview

Before diving into code, let's establish the overall shape of the system. There are two main phases: ingestion (crawling OTCS, chunking, embedding) and retrieval (query processing, hybrid search, answer generation).

Authentication: OTCSTicket

OpenText Content Server uses a session-ticket authentication model. You POST credentials to /api/v1/auth and receive an OTCSTicket that you pass in subsequent requests. The gotcha: tickets expire after a configurable timeout, typically 30–60 minutes. If you're crawling a large repository, you'll hit this mid-operation.

cs_client.py — Authentication with auto-renewal
import requests
import time
from dataclasses import dataclass
from typing import Optional

@dataclass
class CSClient:
    base_url: str
    username: str
    password: str
    _ticket: Optional[str] = None
    _ticket_time: float = 0
    ticket_ttl: int = 1800  # 30 min — refresh before expiry

    def _get_ticket(self) -> str:
        resp = requests.post(
            f"{self.base_url}/api/v1/auth",
            data={"username": self.username, "password": self.password},
            verify=False
        )
        resp.raise_for_status()
        return resp.json()["ticket"]

    @property
    def ticket(self) -> str:
        if (time.time() - self._ticket_time) > self.ticket_ttl:
            self._ticket = self._get_ticket()
            self._ticket_time = time.time()
        return self._ticket

    def get(self, path: str, **kwargs):
        headers = {"OTCSTicket": self.ticket}
        return requests.get(
            f"{self.base_url}{path}",
            headers=headers,
            verify=False,
            **kwargs
        )

The property-based ticket renewal is clean — callers never need to think about auth. Every get() call transparently refreshes if needed.

Node Discovery: Containers vs Documents

This is the most important OTCS-specific concept to understand. In Content Server, not all nodes are documents. Folders, workspaces, categories, and compound documents are "container" nodes. The REST API's children endpoint returns both types, and you need to handle them differently.

⚠

Key fix: The children API response wraps results under a results key (not data). This tripped me up for hours. Always extract response["results"] when iterating children nodes.

cs_client.py — Recursive document discovery
def get_children(self, node_id: int) -> list:
    resp = self.get(f"/api/v2/nodes/{node_id}/nodes")
    resp.raise_for_status()
    data = resp.json()
    # Key insight: children are under "results", not "data"
    return data.get("results", [])

def is_container(self, node: dict) -> bool:
    # container flag is more reliable than node_type alone
    return node.get("data", {}).get(
        "properties", {}
    ).get("container", False)

def walk_nodes(self, root_id: int):
    """Yield all document nodes recursively."""
    stack = [root_id]
    while stack:
        node_id = stack.pop()
        children = self.get_children(node_id)
        for node in children:
            props = node["data"]["properties"]
            if self.is_container(node):
                stack.append(props["id"])
            else:
                yield node  # it's a document

The container flag is more reliable than checking node_type directly — there are several node types (0=folder, 848=workspace, 136=compound doc) that all behave as containers, and the flag covers them all.

Content Extraction

Once you have document nodes, you need their content. For most document types, you can download the file content via the versions API and parse it with the appropriate library (pdfminer for PDF, python-docx for Word, etc.).

extractor.py — Content extraction by MIME type
from io import BytesIO
import pdfminer.high_level as pdf_extract
from docx import Document

def extract_text(content: bytes, mime_type: str) -> str:
    if "pdf" in mime_type:
        return pdf_extract.extract_text(BytesIO(content))
    elif "wordprocessingml" in mime_type:
        doc = Document(BytesIO(content))
        return "\n".join(p.text for p in doc.paragraphs)
    elif "text/plain" in mime_type:
        return content.decode("utf-8", errors="replace")
    else:
        return ""  # skip unsupported types gracefully

Chunking Strategy: Parent-Child

Simple fixed-size chunking loses document structure. The approach I settled on is parent-child chunking: split the document into logical "parent" sections (paragraphs, headings), then create smaller "child" chunks with overlap for embedding. At retrieval time, you retrieve the child chunk but return the parent for LLM context — giving you precision in matching and breadth in context.

chunker.py — Parent-child chunking
from langchain.text_splitter import RecursiveCharacterTextSplitter

parent_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1500,
    chunk_overlap=0,
    separators=["\n\n", "\n", ". "]
)

child_splitter = RecursiveCharacterTextSplitter(
    chunk_size=400,
    chunk_overlap=60,
    separators=["\n", ". ", " "]
)

def create_chunks(text: str, doc_id: str) -> list[dict]:
    chunks = []
    parents = parent_splitter.split_text(text)
    for p_idx, parent in enumerate(parents):
        children = child_splitter.split_text(parent)
        for c_idx, child in enumerate(children):
            chunks.append({
                "doc_id": doc_id,
                "parent_id": f"{doc_id}_p{p_idx}",
                "chunk_id": f"{doc_id}_p{p_idx}_c{c_idx}",
                "child_text": child,   # embedded
                "parent_text": parent  # returned to LLM
            })
    return chunks

Ingestion into ChromaDB

ChromaDB is used here as the vector hot store. Each child chunk gets embedded via Ollama's nomic-embed-text model (768 dimensions) and stored with metadata that enables filtering and parent retrieval. The parent texts are stored separately in a Parquet cold store.

ingester.py — ChromaDB ingestion
import chromadb
from ollama import embeddings

client = chromadb.PersistentClient(path="./chroma_store")
collection = client.get_or_create_collection(
    name="otcs_docs",
    metadata={"hnsw:space": "cosine"}
)

def ingest_chunks(chunks: list[dict]):
    ids, docs, metas, embeds = [], [], [], []
    for chunk in chunks:
        emb = embeddings(
            model="nomic-embed-text",
            prompt=chunk["child_text"]
        )["embedding"]
        ids.append(chunk["chunk_id"])
        docs.append(chunk["child_text"])
        metas.append({
            "doc_id": chunk["doc_id"],
            "parent_id": chunk["parent_id"]
        })
        embeds.append(emb)
    collection.upsert(
        ids=ids, documents=docs,
        metadatas=metas, embeddings=embeds
    )

Hybrid Retrieval with RRF Reranking

Pure vector search misses exact keyword matches — and in enterprise ECM, users often search for specific document IDs, project codes, or contract numbers. Pure BM25 misses semantic similarity. The solution is hybrid retrieval: run both, then fuse the ranked lists using Reciprocal Rank Fusion (RRF).

retriever.py — Hybrid BM25 + Vector with RRF
from rank_bm25 import BM25Okapi
import numpy as np

def rrf_score(ranks: list[int], k: int = 60) -> float:
    return sum(1 / (k + r) for r in ranks)

def hybrid_search(query: str, top_k: int = 5) -> list[dict]:
    # Vector search
    q_emb = embeddings(model="nomic-embed-text", prompt=query)["embedding"]
    vec_results = collection.query(
        query_embeddings=[q_emb], n_results=20
    )
    vec_ids = vec_results["ids"][0]

    # BM25 search over cached corpus
    bm25_scores = bm25.get_scores(query.lower().split())
    bm25_top = np.argsort(bm25_scores)[::-1][:20]
    bm25_ids = [corpus_ids[i] for i in bm25_top]

    # RRF fusion
    scores = {}
    for rank, cid in enumerate(vec_ids):
        scores.setdefault(cid, []).append(rank + 1)
    for rank, cid in enumerate(bm25_ids):
        scores.setdefault(cid, []).append(rank + 1)

    ranked = sorted(scores, key=lambda x: rrf_score(scores[x]), reverse=True)
    return ranked[:top_k]

ACL Security Gates

One thing most RAG tutorials gloss over entirely: in an enterprise system, not every user should see every document. OpenText Content Server has granular ACL-based permissions. Your RAG pipeline needs to enforce these — otherwise you risk surfacing confidential content to unauthorized users.

The clean approach is to store the OTCS node ID with every chunk, and at retrieval time, call GET /api/v2/nodes/{id}/permissions to check whether the current user has read access before returning that chunk to the LLM. Yes, this adds API calls — but correctness matters more than latency for enterprise use.

✓

Optimization: Cache permission checks per user+node pair with a short TTL (5–15 minutes). OTCS permissions rarely change in real-time, and this dramatically reduces API overhead for repeat queries.

Incremental Sync via OTModifyDate

Your knowledge base will go stale without a sync strategy. The key field is OTModifyDate — every OTCS node exposes the last modification timestamp. Track the last successful sync time, and on each run, only fetch nodes where OTModifyDate > last_sync. Combined with the Parquet cold store, this keeps re-embedding costs minimal.

sync.py — Incremental update
from datetime import datetime
import json, pathlib

SYNC_STATE = pathlib.Path("sync_state.json")

def get_last_sync() -> str:
    if SYNC_STATE.exists():
        return json.loads(SYNC_STATE.read_text())["last_sync"]
    return "2000-01-01T00:00:00Z"

def save_sync_time():
    SYNC_STATE.write_text(json.dumps({
        "last_sync": datetime.utcnow().isoformat() + "Z"
    }))

def needs_update(node: dict, last_sync: str) -> bool:
    modify_date = node["data"]["properties"].get("modify_date", "")
    return modify_date > last_sync

Results and Lessons Learned

Running this pipeline against a 500-document OTCS repository, the results are genuinely useful. The system surfaces accurate answers to queries like "what are the contract renewal terms for vendor X" or "find all documents referencing ISO 27001 compliance" — things that previously required hunting through folder hierarchies manually.

A few hard-won lessons:

  • Always use the cold store. Re-embedding 500 documents from scratch costs 20+ minutes. The Parquet cold store paid for itself the first time I switched embedding models.
  • The results key vs data key is a real gotcha. The children endpoint uses results, but single-node endpoints use data. Don't assume consistency.
  • Container detection via the flag, not node_type. I initially checked node_type == 0 for folders, which missed workspace containers (node_type 848). The container boolean flag is the right check.
  • Parent-child chunking over fixed chunking — the retrieval quality difference is significant. Child chunks of ~400 tokens retrieve with precision; parent context of ~1500 tokens gives the LLM enough to reason from.
  • RRF reranking is worth the extra complexity. Hybrid search improved recall on keyword-heavy queries (contract numbers, document IDs) by roughly 40% compared to pure vector search in my testing.
