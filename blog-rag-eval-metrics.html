One Metric to Know If Your OpenText RAG Is Working

Five weeks of architecture decisions — hybrid search, metadata indexing, node ID linkage, caching, ACL enforcement. But how do you know any of it is actually working? Most OpenText RAG pipelines go to production with zero evaluation. That's fine until a compliance question surfaces the wrong document.

Kapil Uthra May 2026 ~11 min read
rag evaluation metrics opentext context-precision

You've built the pipeline. Hybrid search merging BM25 and vector results. Metadata indexed from OpenText categories. Node IDs threaded through every chunk so answers cite back to source documents. Three-layer caching. Group-based ACL enforcement on both retrieval paths. It looks right. It feels right. Users aren't complaining — yet.

Here's the uncomfortable question: how do you actually know it's working? Not "no one has complained" working. Not "it returns something" working. How do you know the right chunks are being retrieved, the answers are faithful to the source, and a restricted user isn't quietly seeing documents they shouldn't?

The answer is evaluation — and the good news is you don't need a full metrics suite to start. You need one metric, run consistently, and one OpenText-specific twist that catches the failure mode most pipelines miss entirely.

Two Failure Modes, One Wrong Answer

Before picking a metric, understand what can actually go wrong. RAG has two distinct failure modes, and conflating them is how teams waste weeks tuning the wrong thing.

Where a Wrong Answer Can Come From
Retrieval Failure
wrong chunks fetched
→
LLM
generates from bad context
→
Wrong Answer
fix: retrieval
Retrieval OK
right chunks fetched
→
Generation Failure
LLM ignores or hallucinates
→
Wrong Answer
fix: prompt / model

A retrieval failure and a generation failure both produce a wrong answer — but the fix is completely different. Tuning your embedding model won't help if the LLM is ignoring the context it's given. Rewriting your prompt won't help if the retrieval pipeline is pulling the wrong chunks. Without measuring both stages separately, you're debugging blind.

Retrieval Metrics

Context Precision — of the chunks you retrieved, how many were actually relevant?

Context Recall — of all relevant chunks in the corpus, how many did you retrieve?

Generation Metrics

Faithfulness — does the answer only claim things supported by the retrieved chunks?

Answer Relevancy — does the answer actually address what was asked?

All four matter. But if you're starting from zero, start with one.

The One Metric: Context Precision

Context precision asks: of the chunks your pipeline retrieved, what fraction were actually useful for answering the question? It's the fastest signal that your retrieval — chunking strategy, embedding model, hybrid fusion weights — is broken. And it's the easiest to calculate without an LLM-as-judge setup.

The formula is simple:

context_precision.py — calculate retrieval quality
from dataclasses import dataclass
from typing import Callable

@dataclass
class EvalCase:
    question: str
    expected_node_ids: list[int]   # OTCS DataIDs of relevant source docs
    retrieved_chunks: list[dict]   # output from your pipeline

def context_precision(
    case: EvalCase,
    relevance_fn: Callable[[dict, EvalCase], bool],
) -> float:
    """
    Average precision @ k — rewards ranking relevant chunks higher.
    A score of 1.0 means all retrieved chunks were relevant.
    A score of 0.3 means most retrieved context is noise.
    """
    hits, precision_sum = 0, 0.0
    for i, chunk in enumerate(case.retrieved_chunks, 1):
        if relevance_fn(chunk, case):
            hits += 1
            precision_sum += hits / i   # precision at position i
    return precision_sum / len(case.retrieved_chunks) if case.retrieved_chunks else 0.0

def node_id_relevance(chunk: dict, case: EvalCase) -> bool:
    """Simple relevance: did we retrieve a chunk from an expected source doc?"""
    return chunk["node_id"] in case.expected_node_ids

# Score across your eval set
def mean_context_precision(cases: list[EvalCase]) -> float:
    scores = [context_precision(c, node_id_relevance) for c in cases]
    return sum(scores) / len(scores)

Notice the relevance function uses node_id — the OTCS DataID threaded through every chunk in Week 3. This is why node ID preservation matters for more than just citations. Without it, you can't programmatically check whether retrieved chunks came from the right source documents. The architecture decisions compound.

✓

What's a good score? A context precision above 0.7 means most retrieved chunks are relevant — your retrieval is working. Between 0.4 and 0.7 is an area worth investigating: look at which query types underperform. Below 0.4 means retrieval is noisy and generation quality is mostly luck.

Building an Eval Set from Real OpenText Queries

The eval set is where most teams cut corners and regret it. A generic eval set built from hypothetical questions tells you nothing about your specific OpenText corpus. You need questions that reflect what your users actually search for — drawn from query logs, user interviews, or your own knowledge of the content.

30–50 question/answer pairs is enough to start. Each case needs:

  1. The question — verbatim or close to how a user would phrase it
  2. The expected source node IDs — which OTCS documents contain the correct answer
  3. Optionally: a reference answer — what the correct response looks like
eval_set.py — structure of an eval case
# Each case maps a real user question to the OTCS docs that answer it
eval_cases = [
    {
        "question": "What is the approval process for capital expenditure above $50,000?",
        "expected_node_ids": [4821993, 4822001],  # Finance Policy v4 + Approval Matrix
        "cohort": "general",                        # which user type should see this
    },
    {
        "question": "Show me all active contracts with Vendor XYZ",
        "expected_node_ids": [3910244, 3910251, 3910288],
        "cohort": "legal",
    },
    {
        "question": "What were the audit findings from Q3?",
        "expected_node_ids": [5103882],
        "cohort": "restricted",                     # only certain groups should access this
    },
]

# Store as JSON — version-control it alongside your pipeline code

The cohort field is the OpenText-specific addition. It tells you which user type each question belongs to — general access, legal team, restricted finance, and so on. That label is what makes the next step possible.

The OpenText Twist: Evaluate Per User Cohort

Here's what generic RAG evaluation frameworks miss entirely. OpenText Content Management pipelines have ACL enforcement — different users can access different documents. If your eval always runs as a single user with broad access, you'll never catch the failure mode that matters most in enterprise: a restricted user retrieving documents they shouldn't see, or failing to retrieve documents they should.

Run your eval set with (at minimum) two user profiles: a general-access user and a restricted user. Track context precision separately for each. If the scores diverge by more than 20 percentage points, something is wrong.

cohort_eval.py — run eval per user ticket
async def run_cohort_eval(
    eval_cases: list[dict],
    pipeline,
    user_tickets: dict[str, str],  # {"general": "ticket_a", "legal": "ticket_b", ...}
) -> dict[str, float]:
    """Run eval set under each user cohort. Returns precision per cohort."""
    results = {}

    for cohort, ticket in user_tickets.items():
        cohort_cases = [c for c in eval_cases if c["cohort"] == cohort]
        if not cohort_cases:
            continue

        scores = []
        for case in cohort_cases:
            # Run the real pipeline — with real ACL enforcement
            retrieved = await pipeline.retrieve(
                query=case["question"],
                user_ticket=ticket,
            )
            eval_case = EvalCase(
                question=case["question"],
                expected_node_ids=case["expected_node_ids"],
                retrieved_chunks=retrieved,
            )
            scores.append(context_precision(eval_case, node_id_relevance))

        results[cohort] = sum(scores) / len(scores)
        print(f"cohort={cohort}  precision={results[cohort]:.2f}  n={len(scores)}")

    return results

# Example output you want to see:
# cohort=general    precision=0.82  n=30
# cohort=legal      precision=0.79  n=12
# cohort=restricted precision=0.81  n=8
#
# Example output that flags a problem:
# cohort=general    precision=0.83  n=30
# cohort=restricted precision=0.51  n=8  ← >20% divergence — ACL bug

A >20% divergence between cohorts is a red flag that your ACL enforcement is inconsistent. Either the index was built with incorrect group tags, the group resolver is returning the wrong memberships for certain ticket types, or the metadata store path isn't applying the same filter as the vector store. Any of these will silently degrade the experience for restricted users — the exact people the ACL system was supposed to protect.

⚠

Test with your highest-access users too. A VP or records manager with access to everything is the canary for retrieval quality unconstrained by ACL. If their precision is dramatically higher than a general user on the same questions, your ACL filtering may be cutting too aggressively — discarding relevant chunks that should be accessible.

The Full Metrics Picture

Once context precision is stable and you're running it per cohort, add the other three metrics one at a time. Each one requires a bit more infrastructure but tells you something the others can't.

Metric Stage What it catches How to calculate
Context Precision Retrieval Noisy retrieval — irrelevant chunks crowding out relevant ones Average precision @ k against expected node IDs
Context Recall Retrieval Missed retrieval — relevant chunks exist in corpus but weren't fetched Fraction of expected node IDs present in retrieved set
Faithfulness Generation Hallucination — answer makes claims not supported by retrieved chunks LLM-as-judge: does each claim in the answer appear in context?
Answer Relevancy Generation Off-topic answers — LLM answered a different question than was asked LLM-as-judge or embedding cosine similarity of Q vs answer

Context recall is the natural complement to precision. Precision says "are the chunks you fetched good?" Recall says "are you missing good chunks?" A pipeline with high precision but low recall is finding relevant content when it retrieves anything, but often not retrieving enough — a chunking or embedding problem. A pipeline with high recall but low precision is casting a wide net and letting the LLM sort it out — a fusion or re-ranking problem.

ℹ

Faithfulness and answer relevancy both need an LLM judge, which adds cost and latency to your eval pipeline. Start with the retrieval metrics (precision and recall), which are purely deterministic and cheap. Add the generation metrics once retrieval is stable — otherwise you're evaluating a moving target.

Make It a CI Gate, Not a Launch Checklist

Evaluation run once at launch is nearly useless. The OpenText corpus changes. Users add documents, restructure workspaces, update categories. Your pipeline evolves — chunking strategy, fusion weights, a model swap. Each change can silently degrade retrieval quality, and you won't know until a user surfaces the wrong document in a compliance review.

The right cadence is to run your eval set on every significant pipeline change and on a weekly schedule regardless of changes. That weekly run is your drift detector.

eval_runner.py — weekly scheduled eval with drift detection
import json, datetime

async def run_weekly_eval(pipeline, eval_cases, user_tickets, store):
    results = await run_cohort_eval(eval_cases, pipeline, user_tickets)
    timestamp = datetime.datetime.utcnow().isoformat()

    # Persist results — compare against previous run
    previous = store.get_latest()
    entry = {"timestamp": timestamp, "scores": results}
    store.save(entry)

    if previous:
        for cohort, score in results.items():
            prev_score = previous["scores"].get(cohort, score)
            delta = score - prev_score
            if delta < -0.10:  # >10% drop vs last week
                alert(f"PRECISION DROP: {cohort} {prev_score:.2f} → {score:.2f} ({delta:+.2f})")

    return results

# Structural changes that should trigger an immediate eval run:
# - new chunking strategy or chunk size
# - changed embedding model
# - modified hybrid fusion weights
# - OpenText Category restructure (changes metadata store records)
# - new ACL group added or permission scope changed

A 10% precision drop week-over-week is a meaningful signal. It doesn't mean something catastrophically broke — it might be a new batch of documents with different formatting, or a category rename that broke your metadata filters. But it's worth investigating before users notice.

The Discipline Matters More Than the Dashboard

This series started with hybrid search — the architecture decision that makes retrieval work across structured and unstructured queries. It ends with evaluation — the discipline that tells you whether the architecture is actually delivering.

You don't need a full observability platform to start. You need:

  • 30–50 real queries drawn from your OpenText use case, tagged with expected source node IDs
  • Context precision calculated against that ground truth, run per user cohort
  • A weekly scheduled run with a simple drift alert

That's enough to catch the most common failures — noisy retrieval, ACL inconsistency, and silent degradation as the corpus evolves. Everything else is refinement on top of that foundation.

Measure context precision. Run it per cohort. Alert on drift. That's the evaluation habit that turns an OpenText RAG pipeline from "it seems to work" into "we know it works."
