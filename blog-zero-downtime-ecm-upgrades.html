Zero-Downtime ECM Upgrades: Lessons from KSA, Brazil, and Finland

Planning and executing OpenText Content Server major version upgrades with minimal outage windows — drawn from 10+ upgrades across three continents, including a 90% outage reduction for a US manufacturing client and a GxP-constrained upgrade for a Finnish pharmaceutical company.

Kapil Uthra January 2026 ~13 min read
upgradesopentextoperationsawsenterprise

Content Server upgrades have a reputation for being painful. That reputation is earned — but mostly for the wrong reasons. The technical upgrade itself is well-documented and generally reliable. The pain comes from customization debt, integration regression, and insufficient pre-upgrade testing. I've led major version upgrades across 10+ enterprise deployments, and the pattern is consistent: the projects that go smoothly are the ones that invest heavily in preparation, not in hoping the upgrade script works.

Let me walk through what a well-executed upgrade looks like, with examples from three specific engagements that presented different constraints.

The Upgrade Preparation Checklist

The single biggest lever on upgrade success is how thoroughly you prepare. A minimum viable preparation for a major version upgrade (e.g., 16.2 → 22.4) should cover:

T-8 wk
Customization Inventory
Enumerate every custom module, web service, workflow, and WebReports script. Document the version compatibility status of each against the target version. This is the work no one wants to do but that determines whether the upgrade is 2 hours or 2 weeks.
T-6 wk
Integration Regression Plan
List every system that integrates with OTCS: SAP, Salesforce, archiving systems, Brava/IV viewers, OTDS, Archive Center. Write a test case for each integration. You'll need to execute these on staging before go-live.
T-4 wk
Staging Environment Upgrade
Run the full upgrade procedure on a staging environment that mirrors production as closely as possible. This isn't a rehearsal — treat it as the real thing. Document every error and resolution.
T-2 wk
Rollback Plan + Dry Run
Define the exact rollback procedure. Test it on staging: upgrade, verify, rollback, verify rollback. If your rollback procedure doesn't work on staging, it won't work at 2am in production.
T-1 wk
Communication + Runbook
Publish the maintenance window to all users. Distribute the detailed runbook to the upgrade team with every step, expected duration, and success criteria written out.

The SABIC Case: Tight Outage Window

SABIC (KSA) had a strict constraint: the Content Server upgrade had to complete within a 4-hour outage window, during which chemical plant operations documentation would be inaccessible. They had a complex customization landscape and were upgrading from 10.5 to 16.2 — a significant version jump with multiple intermediate schema changes.

The approach that worked: pre-stage everything offline before the maintenance window begins. We stood up a parallel environment with the new OTCS version pre-installed, all modules pre-loaded, and all customizations pre-tested. During the maintenance window, the only work that happened in production was: (1) stop services, (2) run database schema upgrade scripts, (3) copy the pre-staged CS binaries over the existing installation, (4) update configuration, (5) restart. The 4-hour window became 2.5 hours.

We partnered directly with OpenText engineering for the schema migration scripts — at 16.2 there were non-trivial schema changes that required OpenText's involvement to execute correctly. That relationship matters: for complex upgrades, escalating early to OpenText support is not admitting failure, it's good risk management.

USA — Manufacturing: 90% Outage Reduction

SCG ran a 30+ server AWS landscape — a distributed Content Server deployment with multiple front-end nodes, separate search/indexing nodes, and an Archive Center integration. Their historical upgrade outages ran 8–12 hours. The goal was to get under 1 hour.

The key change was upgrading the architecture to support rolling upgrades. In a single-server deployment, you can't do rolling upgrades — you take the server down, upgrade it, bring it back up. In a distributed deployment, you can take one front-end node out of the load balancer, upgrade it, verify it, put it back, then move to the next node. Users continue to use the remaining nodes during the process.

The real work happened before the maintenance window. Extensive prep and multiple dry-run rounds on a staging environment that closely mirrored production let us understand exactly how long every step would take and where the risks sat. We also fast-tracked two things that are often left until upgrade day: the database upgrade (schema changes applied and validated in advance on a restored production snapshot) and the full-text indexing rebuild (pre-staged so the index wasn't starting from zero after go-live). By the time the production window opened, the team had executed the same sequence four times. There were no surprises.

Result: 30+ servers upgraded with a total user-facing outage of under 45 minutes — only the unavoidable database schema step required all nodes to be stopped simultaneously. That's the 90% reduction.

Finland — Pharmaceuticals: GxP Constraints

Pharmaceutical environments under GxP regulations have special upgrade requirements. Every change to a validated system must go through Change Control, have documented impact assessment, require User Acceptance Testing with sign-off, and have an audit trail of the upgrade execution itself.

The technical upgrade was straightforward. The GxP overhead was not. Here's what the process required on top of the standard upgrade:

  • Change Control documentation including risk assessment, impact analysis, and proposed validation approach — submitted and approved 6 weeks in advance.
  • IQ/OQ/PQ protocols (Installation Qualification, Operational Qualification, Performance Qualification) written, executed, and signed off for the new version.
  • Validation traceability matrix mapping every test case to a regulatory requirement.
  • Post-upgrade audit trail showing who performed each upgrade step, at what time, with what result — captured in the system's own audit log where possible.
✓

GxP tip: use the Content Server audit log itself as part of your post-upgrade IQ documentation. The first entries after upgrade start should be logins by the validation team executing IQ scripts — this gives you a timestamped record in the validated system's own audit trail.

This upgrade was technically the simplest of the three I've described here. The discipline required for GxP compliance makes it the most time-consuming — but that discipline is the point. In a pharmaceutical environment, the document management system is part of the product quality infrastructure. It should be treated accordingly.

Common Failure Modes

Across 10+ upgrades, the failures cluster around the same root causes:

  • Undocumented customizations. Something breaks after upgrade. The fix is a module or script that no one documented and that turns out to be incompatible with the new version. This one is entirely preventable with a proper customization inventory.
  • Database maintenance skipped pre-upgrade. Index fragmentation and stale statistics cause the schema upgrade scripts to run slower than expected, blowing the maintenance window. Always run a full index rebuild and statistics update 48 hours before the upgrade window.
  • Rollback procedure never tested. Rollback is needed, the procedure is unclear, hours are lost. Test your rollback on staging.
  • Integration testing done in production. The first test of the SAP or Salesforce integration after upgrade happens in the production environment. Problems that would have been caught and fixed on staging cause go-live delays.

None of these failures are interesting. They're all preventable. The preparation phase is where upgrades succeed or fail — the maintenance window is just execution.
