How I Debugged 150M Document Download Failures in cs_client.py

The container flag vs node_type trap, the results vs data key mismatch, and the diagnostic scripts that finally cracked it — an honest account of two days of head-scratching with the OpenText REST API.

Kapil Uthra March 2026 ~12 min read
debuggingopentextpythonapirag

Two days. That's how long it took me to track down two bugs in cs_client.py that were silently causing document download failures when I ran my RAG ingestion pipeline against an OpenText Content Server repository. Not catastrophic failures — the kind that quietly skip documents and leave you wondering why your vector store is half-empty.

This is the story of those two bugs, how I found them, and what the fixes look like. If you're building anything against the OTCS REST API, these will save you the same two days.

✗

Symptom: RAG pipeline completes without error but vector store contains far fewer documents than expected. Logs show successful crawl but zero content extracted from what should be document nodes.

Bug 1: The Container Flag

The first bug was in how I detected container nodes (folders, workspaces, compound documents) versus leaf document nodes. My original code checked the node_type field directly:

- # Original — only catches folders (type 0)
- def is_container(node): - return node["data"]["properties"]["node_type"] == 0
+ # Fixed — uses the container boolean flag
+ def is_container(node): + return node["data"]["properties"].get("container", False)

Content Server has multiple node types that behave as containers: 0 (Folder), 848 (Workspace), 136 (Compound Document), 751 (Project), and others. My node_type == 0 check was treating workspace nodes (848) as leaf documents — and then trying to download their "content", which of course returned nothing or errored silently.

The container boolean in the properties object is the correct check. It's maintained by OTCS regardless of the specific type, and it's exactly what the UI uses internally to decide whether to show expand/collapse controls.

Once I fixed this, the traversal correctly descended into workspace hierarchies that had previously been misidentified as documents.

Bug 2: The results vs data Key

The second bug was subtler — and frankly, it looks like an inconsistency in the OTCS API itself rather than user error.

When you call the single-node endpoint GET /api/v2/nodes/{id}, the response is structured as:

Single node response structure
{
  "data": {
    "properties": { "id": 12345, "name": "Contract.pdf", ... }
  }
}

But when you call the children endpoint GET /api/v2/nodes/{id}/nodes, the response is structured differently:

Children endpoint response structure
{
  "results": [              // NOT "data"
    {
      "data": {
        "properties": { "id": 12346, ... }
      }
    },
    ...
  ]
}

My original code was doing response.json().get("data", []) on the children response, which returned an empty list every single time. The traversal thought every folder had zero children and stopped immediately.

- children = resp.json().get("data", [])
+ children = resp.json().get("results", [])

One character difference. Two days of debugging. Such is the life.

The Diagnostic Script That Found It

What eventually cracked both bugs was writing a targeted diagnostic script rather than staring at the main pipeline code. The key insight was to isolate the API calls completely and print the raw response structure:

diagnostic.py — Raw API response inspector
import json
import requests

def inspect_node(base_url, ticket, node_id):
    headers = {"OTCSTicket": ticket}

    # Inspect single node
    r1 = requests.get(f"{base_url}/api/v2/nodes/{node_id}", headers=headers, verify=False)
    print("=== Single node top-level keys ===")
    print(list(r1.json().keys()))

    # Inspect children
    r2 = requests.get(f"{base_url}/api/v2/nodes/{node_id}/nodes", headers=headers, verify=False)
    print("=== Children top-level keys ===")
    print(list(r2.json().keys()))

    # Print first child structure
    children_data = r2.json()
    for key in ["results", "data"]:
        if key in children_data and children_data[key]:
            first = children_data[key][0]
            print(f"\nFirst child via '{key}':")
            props = first.get("data", {}).get("properties", {})
            print(f"  name: {props.get('name')}")
            print(f"  node_type: {props.get('node_type')}")
            print(f"  container: {props.get('container')}")

Running this against a known folder node immediately showed me that "data" was at the top level of single-node responses but "results" was the children key — and printing the container property alongside node_type made the workspace misclassification obvious.

Why These Bugs Are Easy to Make

Neither of these is an obvious mistake if you're working from the OTCS REST API documentation alone. The API reference describes the node types numerically without making it clear that the container flag is the canonical way to test for container behaviour. And the inconsistency between data and results as top-level keys across different endpoints isn't flagged anywhere — you'd only notice it if you read two different endpoint docs side by side and thought to compare them.

Having spent 13 years with Content Server, I knew the platform well enough to be surprised — which probably made me look in the wrong places first. Sometimes deep familiarity with a system makes you assume things work the way they used to, even when the REST API is a relatively newer layer on top of older behaviour.

The Fixed cs_client.py

cs_client.py — Both fixes applied
def get_children(self, node_id: int) -> list:
    resp = self.get(f"/api/v2/nodes/{node_id}/nodes")
    resp.raise_for_status()
    # Fix 2: use "results" key, not "data"
    return resp.json().get("results", [])

def is_container(self, node: dict) -> bool:
    # Fix 1: use container flag, not node_type == 0
    return node.get("data", {}).get(
        "properties", {}
    ).get("container", False)

def walk_nodes(self, root_id: int):
    """Yield all document nodes recursively."""
    stack = [root_id]
    while stack:
        node_id = stack.pop()
        children = self.get_children(node_id)
        for node in children:
            if self.is_container(node):
                props = node["data"]["properties"]
                stack.append(props["id"])
            else:
                yield node

With both fixes in place, the ingestion pipeline correctly traversed the full node hierarchy and extracted content from all document nodes. The vector store went from partially populated to complete, and RAG query quality improved substantially.

If you're hitting similar issues, run the diagnostic script first — five minutes of raw API inspection beats hours of staring at application code.
