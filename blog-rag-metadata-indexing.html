What to Index from OpenText Content Server — and What to Leave Out

Not all metadata is retrieval-useful. Indexing everything from OTCS bloats your vector store, slows queries, and adds noise. Here's a field-by-field breakdown of what actually improves RAG quality.

Kapil Uthra May 2026 ~11 min read
rag metadata opentext otcs indexing

The naive approach to building a RAG pipeline on top of OpenText Content Server goes like this: pull every document, extract its text, pull all the metadata from the node properties endpoint, concatenate everything into a chunk, embed it, and store it. Job done. Ship it.

The result is a vector store that's 40% larger than it needs to be, with embedding quality degraded by metadata noise, retrieval that surfaces documents from the wrong lifecycle stage, and an LLM context window cluttered with internal system IDs that mean nothing to the model. The answers look plausible but aren't well-grounded — and the errors are subtle enough to slip through review.

Metadata design for RAG is not a minor implementation detail. It is the retrieval configuration layer — it determines which documents can be filtered, how the LLM interprets document context, and whether citations are meaningful. This is the field-by-field breakdown I wish existed when I started building OTCS-connected RAG systems.

Why Metadata Matters Beyond the Document Body

In a pure embedding pipeline, retrieval is purely semantic: you embed the query, find the nearest chunks by cosine similarity, and send the top-k to the LLM. Metadata doesn't factor in at all. This works acceptably for small corpora of uniform, current documents.

In enterprise ECM, this breaks in three specific ways:

  • Version confusion. Your vector store has five versions of the same safety policy. Without version metadata, semantic search will retrieve whichever version has the highest embedding similarity to the query — which may be the 2018 draft, not the 2024 approved version.
  • Workspace bleed. Documents from the HR workspace and the Finance workspace can have nearly identical semantic content (both discuss "budget approval thresholds"). Without workspace metadata as a filter, a Finance user's query retrieves HR documents they may not be authorised to see — or simply doesn't have.
  • Stale document retrieval. OTCS repositories accumulate superseded documents. Expired contracts, retired procedures, cancelled projects — they're all still in the repository, searchable, embeddable. Without a "Status" metadata filter, your RAG will confidently cite a policy that was withdrawn three years ago.

Metadata filters pre-restrict the retrieval candidate set before embedding comparison even begins. And metadata stored alongside chunks provides context the LLM uses when constructing its answer: "this is an expired contract from 2019" changes how the model should qualify its response.

The OTCS Node Metadata Landscape

Every node in OpenText Content Server has a standard set of system properties accessible via GET /api/v2/nodes/{id}. Understanding what each field actually represents is the prerequisite for deciding what to index.

The standard node properties response includes:

  • id (DataID) — the integer node identifier, permanent and unique
  • parent_id — the DataID of the parent folder or workspace
  • name — the display name of the document
  • type — node type integer (144 = document, 0 = folder, 848 = workspace)
  • create_date, modify_date — ISO 8601 timestamps
  • create_user_id, modify_user_id — integer user IDs
  • mime_type — for documents (e.g., application/pdf)
  • version — version number of the current version
  • versions_control_advanced — whether versioning is enabled
  • description — optional free-text description field

Beyond system properties, each node can have Categories — custom metadata attribute sets attached at the workspace or document level. Categories are accessed via GET /api/v2/nodes/{id}/categories and are the most powerful and most underused feature in OTCS-connected RAG implementations.

What TO Index — and Why

DataID (node_id) — mandatory

Store this as a structured metadata field on every chunk, never embedded in the text. It is your permanent linkback key to the source document. Without it, your citations are decoration. Every vector store supports scalar metadata fields — use one for node_id. This is covered in depth in the node ID preservation article, but the short version is: if you don't store it now, you will rebuild your index later to add it.

Name — critical signal

Document names in OTCS carry dense, structured signal. "Annual EHS Report 2023 v4 APPROVED" tells you: domain (EHS), document type (report), year (2023), version (4), and status (approved) — all from the name field. Including this in the embedded text of every chunk significantly improves retrieval relevance for queries that reference document type or time period.

Prepend the name to each chunk at index time. Don't rely on users discovering it through semantic similarity alone.

ModifyDate — essential for recency filtering

Store as an ISO date string or Unix timestamp metadata field. Use it to build pre-retrieval filters for queries where currency matters. "What is our current policy on X?" should filter to documents modified within the last two years, not surface the 2015 version that happens to have the highest embedding similarity.

ModifyDate also powers staleness warnings: if a chunk was indexed six months ago and the document was modified last week, surface a flag to the user.

VersionNumber — index as metadata, not embedded text

Store version as a structured metadata field (integer or string). Do not include it in the embedded text chunk. Version numbers are used as filters, not as semantic content — embedding them adds noise to the vector representation without adding retrieval value. Use the metadata field to filter to the latest version when that's the intent, or to a specific version when auditing.

Category attributes — selective, high-signal fields only

This is where the real value lies. OTCS categories are custom attribute sets defined at the workspace or repository level. Common high-signal attributes in enterprise deployments include:

  • Document Class — "Policy", "Procedure", "Contract", "Report", "Form"
  • Status / Lifecycle State — "Draft", "Under Review", "Approved", "Superseded", "Expired"
  • Contract Type — "NDA", "MSA", "SOW", "Lease"
  • Applicable Region / Jurisdiction — for multi-region organisations
  • Confidentiality Level — "Public", "Internal", "Confidential", "Restricted"

Index these as structured metadata fields. They are filters, not semantic content — but including the values in the chunk text header (e.g., "Document Class: Policy | Status: Approved") also helps the LLM contextualise the document when generating answers.

Parent workspace name — locality context

Resolve the parent workspace name at index time (via GET /api/v2/nodes/{parent_id}) and store it as metadata. "This chunk is from the HR Policies Workspace" vs "this chunk is from the Finance Procedures Workspace" is context the LLM needs to qualify its answers accurately. It also enables workspace-scoped retrieval without needing a separate index per workspace.

MimeType — for filtering, not embedding

Store as a metadata field. Use it to exclude document types that produce poor retrieval results: scanned image PDFs with no OCR text, .msg email attachments, legacy binary formats. A filter for mime_type IN ["application/pdf", "application/vnd.openxmlformats-officedocument.wordprocessingml.document"] restricts retrieval to document types your text extraction pipeline can actually handle well.

What NOT to Index — and Why

ParentID as a number — meaningless noise

The raw parent_id integer (e.g., 3847291) means nothing to an embedding model and nothing to the LLM. If you need workspace context, resolve it to the workspace name at index time. The raw ID adds noise to embeddings and bloats your metadata payload without any retrieval or generation value.

CreatedBy / ModifiedBy as raw user IDs

OTCS user IDs are integers. Embedding "modified_by": 10482 in your chunk text tells the LLM nothing. If author information is relevant to your use case (e.g., "find all documents written by the legal team"), resolve user IDs to display names using GET /api/v2/members/{id} at index time and store the resolved name. Raw user IDs are noise.

⚠

Privacy consideration: Even resolved user names can be sensitive. In some jurisdictions, indexing "this document was last modified by [Name]" into a system that generates AI responses constitutes personal data processing. Check your data privacy requirements before indexing author information at all.

VersionComment — almost always empty or noise

OTCS prompts users for a version comment when checking in a new version. In practice, these are almost universally empty, "minor fix", "updated per review", or "see email from 14/03". They're not semantic content — they're workflow artefacts. Index them only if your organisation has a disciplined practice of writing meaningful version comments, which is rare.

All category attributes blindly

A mature OTCS deployment will have 30–60 custom category attributes defined across different workspace templates. Most of these are workflow state fields: "Assigned To", "Review Due Date", "Approval Status", "Workflow Instance ID". These are operational metadata for the OTCS workflow engine — not semantic signal for RAG retrieval.

Indexing all of them bloats your embedded chunk text with values like "Workflow Instance ID: 8473920" and "Review Due Date: 2024-03-15" — both of which contaminate your embedding with tokens that look like dates and IDs but contribute nothing to semantic similarity. Limit your category indexing to 5–8 attributes that describe the document's purpose, domain, and audience.

Full ACL permission lists — never

Never embed the ACL. Not as text, not as metadata visible to the LLM. ACLs contain user IDs and group IDs that are sensitive system information, and including them in chunk text means the LLM can potentially surface them in answers. Store a node_id and resolve permissions at query time via the permission resolver — that's the right architecture, covered in the ACL-aware RAG article.

Chunking Strategy for OTCS Documents

The metadata you've selected needs to travel with every chunk through the pipeline. The most common failure mode is chunking a document into text fragments and then losing the metadata association — the chunk knows its text, but not which node it came from.

Two chunking strategies that work well for OTCS documents:

  • Heading-based chunking — for structured documents (policies, procedures, technical manuals). Split at H1/H2 headings, preserving section context. Each chunk is semantically coherent and maps to a document section.
  • Fixed-window with overlap — for unstructured documents (contracts, reports). 512-token windows with ~10% overlap (50 tokens). Overlap prevents important content from being split exactly at a chunk boundary.

In both cases, prepend a metadata header to every chunk. This header is embedded with the text — it gives the embedding model context about the document type and status, which improves retrieval precision for filtered queries.

chunk_builder.py — chunk with metadata header, preserving node linkage
from dataclasses import dataclass
from typing import Optional
import textwrap

@dataclass
class OTCSChunk:
    node_id: int
    node_name: str
    workspace_name: str
    doc_class: str
    status: str
    modify_date: str
    version: int
    mime_type: str
    chunk_index: int
    text: str          # raw chunk text, no header
    embed_text: str    # header + text, used for embedding

def build_chunk(
    node_id: int,
    node_name: str,
    workspace_name: str,
    doc_class: str,
    status: str,
    modify_date: str,
    version: int,
    mime_type: str,
    chunk_index: int,
    chunk_text: str,
) -> OTCSChunk:
    # Metadata header prepended to every chunk for embedding
    header = (
        f"Document: {node_name}\n"
        f"Workspace: {workspace_name}\n"
        f"Class: {doc_class} | Status: {status} | Modified: {modify_date}\n"
        f"---\n"
    )
    return OTCSChunk(
        node_id=node_id,
        node_name=node_name,
        workspace_name=workspace_name,
        doc_class=doc_class,
        status=status,
        modify_date=modify_date,
        version=version,
        mime_type=mime_type,
        chunk_index=chunk_index,
        text=chunk_text,
        embed_text=header + chunk_text,
    )

def to_vector_store_doc(chunk: OTCSChunk) -> dict:
    """Convert chunk to vector store ingestion format."""
    return {
        "id": f"{chunk.node_id}_{chunk.chunk_index}",
        "text": chunk.embed_text,   # embed the header+text
        "metadata": {
            "node_id": chunk.node_id,
            "node_name": chunk.node_name,
            "workspace": chunk.workspace_name,
            "doc_class": chunk.doc_class,
            "status": chunk.status,
            "modify_date": chunk.modify_date,
            "version": chunk.version,
            "mime_type": chunk.mime_type,
            "chunk_index": chunk.chunk_index,
        }
    }

The Category Attribute Opportunity

OTCS categories are the hidden gem that most RAG implementations ignore. A category is a named set of typed attributes attached to a workspace or document type. When a document is in the "Contract Management" workspace, it might have a "Contract Details" category with attributes like Contract Type, Counterparty, Effective Date, Expiry Date, and Contract Status.

These attributes are retrieved via GET /api/v2/nodes/{id}/categories, which returns a structured JSON object with all category definitions and their current values for that node. The response structure looks like:

OTCS categories API response — example structure
# GET /api/v2/nodes/4821993/categories
{
  "data": {
    "categories": {
      "17429": {           # category node ID
        "17429_2": "NDA",              # Contract Type
        "17429_3": "Acme Corp Ltd",   # Counterparty
        "17429_4": "2024-01-15",       # Effective Date
        "17429_5": "2026-01-14",       # Expiry Date
        "17429_6": "Active"            # Contract Status
      }
    }
  }
}

To use these effectively you need to resolve the attribute IDs (17429_2) to human-readable names using the category definition endpoint GET /api/v2/nodes/{category_id}. Build a category schema map at startup and reuse it for all documents in the same workspace.

Once resolved, a Contract Status = Active attribute becomes a first-class metadata filter. Your retrieval query can include metadata.contract_status = "Active" as a hard pre-filter, eliminating expired and draft contracts from retrieval entirely. This is functionality that most RAG implementations simulate with post-hoc LLM filtering — which is slower, less reliable, and doesn't actually prevent expired content from entering the context window.

✓

Build a category registry: For each workspace type in your OTCS deployment, map the category attribute IDs to their human-readable names and data types once, at pipeline setup time. Store this mapping in your pipeline config. The attribute IDs are stable — they don't change unless an administrator modifies the category definition, which is rare and auditable.

Metadata Is Your Retrieval Configuration

The documents in your vector store are fixed once indexed. The metadata you index with them determines every filter, every scope constraint, every staleness check, and every piece of LLM context that shapes how answers are generated. Getting it wrong means rebuilding your index — which in a 200,000-document OTCS deployment is a multi-day operation.

Design your metadata schema before you index a single document. Ask: what filters will users need? What context does the LLM need to qualify its answers accurately? What fields do you need to build the linkback URL? What fields signal document currency vs. staleness?

Treat metadata design like API design — it's a contract between your indexing pipeline and your retrieval layer. Change it carefully, version it explicitly, and document the rationale for every field you include and every field you deliberately exclude.
