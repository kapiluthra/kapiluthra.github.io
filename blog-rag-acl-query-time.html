ACL Enforcement in OpenText RAG: Protecting Both Retrieval Paths

Hybrid search means two retrieval paths — vector for semantic, keyword for exact match. For enterprise security, ACLs must live in both stores and filter independently before results are merged. One unprotected path is all it takes for a permission leak.

Kapil Uthra May 2026 ~12 min read
rag acl security opentext hybrid-search permissions

Your RAG pipeline runs hybrid search — vector for semantic matching, keyword for exact lookups — and you've tested it. It works. Then a VP with access to a million documents runs a query. One of two things happens: the system times out trying to filter the vector store by 800,000 node IDs, or the keyword path silently returns documents that user shouldn't see because the metadata store filter wasn't wired up. Either way, you have a production incident.

The fix isn't to optimise the document list lookup. It's to filter on the right dimension — group IDs, not document IDs — and to do it on both retrieval paths before results are merged.

Why the Naive Pattern Fails at Scale

The intuitive ACL pattern:

  1. User submits a query
  2. Resolve all document IDs the user can access (via OTCS REST or a permission table)
  3. Pass that list as a node_id IN [...] filter to the vector store and/or metadata store
  4. Run search within the filtered set

For a user with access to 500 documents, this works. For a user with access to 500,000 documents, you're passing a filter clause with half a million integer IDs. Most vector stores have limits on filter clause size. And even if your keyword store accepts it, the query planner still has to evaluate that predicate on every candidate row. The failure mode is entirely predictable — it's just a matter of when a high-access user shows up.

❌ Naive — filter by document list

user has access to 800k docs
→ filter: node_id IN [1,2,3...800000]
→ times out / malformed clause
→ both retrieval paths break

✓ Correct — filter by group (both stores)

user belongs to 12 groups
→ filter: acl_group_id IN [g1,g2...g12]
→ fast, bounded, stable
→ works on vector + metadata store

The Architectural Insight: Dual-Store ACL with Group Filtering

OpenText Content Management permissions are group-based by design. A user belongs to a small number of groups — typically 5 to 50. Those groups grant access to potentially millions of documents. The asymmetry is the key: the user's group set is tiny and stable; their document set is large and dynamic.

The correct architecture exploits this asymmetry across both retrieval paths. At index time, tag every chunk with acl_group_id in both the vector store (for semantic search) and the metadata store (for exact-match and attribute queries). At query time, resolve the user's group memberships once — cached per session — and pass those same group IDs as the filter to whichever path runs. Both paths return permission-safe results. Reciprocal Rank Fusion merges them. No path has to trust the other.

Index Time: Store acl_group_id in Both Stores

The key indexing decision: acl_group_id must land in both the vector store and the metadata store. Each retrieval path needs to enforce ACL independently — if one store is missing the field, that path becomes a permission leak.

Dual-store indexing — acl_group_id in both records
# Vector store record (lean — for similarity search + group filter)
vector_store_record = {
    "id": "chunk_4821993_0",
    "embedding": [...],            # float vector
    "node_id": 4821993,            # OTCS DataID — link back to source
    "acl_group_id": "grp_legal",   # group that grants access — filter key
    "chunk_index": 0,
}

# Metadata store record (keyed by node_id — for exact match + attribute queries)
metadata_store_record = {
    "node_id": 4821993,
    "name": "Contract Template v3.2",
    "modify_date": "2026-03-14",
    "version": 3,
    "category": "Active Contract",
    "workspace": "Legal Workspace",
    "acl_group_id": "grp_legal",   # same field — enables keyword path ACL filter
    "full_acl": {"grp_legal": "SeeContents", "grp_admin": "Full"},
}

The vector store handles semantic similarity and group-filtered search. The metadata store handles exact lookups, attribute filters, and keyword queries — queries like "CV-2024-001" or "all Q3 Finance contracts" that don't belong in vector similarity at all (that's what Week 1 was about). Both stores carry acl_group_id; both filter on it before returning results. Neither has to trust the other's output.

ℹ

What if a document is accessible to multiple groups? Store one record per group that grants access — same chunk, same embedding, different acl_group_id. Do this in both stores. It increases index size slightly but keeps the query-time filter trivially simple. The alternative (storing a list of group IDs per record) requires stores that support array membership filters, which not all do.

Query Time: Resolve Groups, Filter, Re-Check

The query path has two stages. Stage 1 is fast and handles the bulk of filtering. Stage 2 is a targeted safety net for edge cases.

Stage 1 — Vector Search with Group Filter

acl_retrieval.py — group-based filtered search
async def retrieve_with_acl(
    query_embedding: list[float],
    user_ticket: str,
    group_resolver,   # cached OTCS group lookup
    vector_store,
    top_k: int = 20,
) -> list[dict]:

    # Resolve user's groups — cached per session, typically 5–50 IDs
    user_groups = await group_resolver.get_groups(user_ticket)

    if not user_groups:
        return []  # no group memberships → no access

    # Stage 1: vector search with group filter (fast — bounded filter clause)
    results = await vector_store.search(
        embedding=query_embedding,
        filter={"acl_group_id": {"$in": user_groups}},
        k=top_k,
    )
    return results

Stage 2 — Per-Document Re-Check on Top-K Only

Group-based filtering handles the common case. But OpenText supports document-level permission overrides — a user may be in a group that grants broad access but be explicitly denied on a specific node. Stage 2 catches these edge cases by doing a targeted ACL re-check on just the top-k results, not the whole corpus.

Stage 2 — fine-grained per-document verification
async def verify_top_k(
    results: list[dict],
    user_ticket: str,
    otcs_client,
) -> list[dict]:
    """Re-check See Contents permission on each top-k node."""
    verified = []
    for chunk in results:
        node_id = chunk["node_id"]
        # HEAD request to node content endpoint — 200 = accessible, 403 = denied
        ok = await otcs_client.can_see_contents(user_ticket, node_id)
        if ok:
            verified.append(chunk)
    return verified

# Full pipeline
async def acl_aware_retrieve(query_embedding, user_ticket, ...):
    stage1 = await retrieve_with_acl(query_embedding, user_ticket, ...)
    stage2 = await verify_top_k(stage1, user_ticket, otcs_client)
    return stage2

Stage 2 makes at most top_k OTCS API calls — typically 10–20. That's bounded and fast, unlike the naive pattern which makes unbounded calls proportional to the user's total accessible set.

The Metadata Store Path: Exact Match with the Same ACL Key

Vector search handles "find documents similar to this concept." The metadata store handles something different: "find the document named CV-2024-001" or "list all Q3 Finance contracts approved after June." These are exact-match and attribute queries — they should hit the metadata store directly with SQL-style filters, not be forced through vector similarity.

The ACL filter here uses the same key — acl_group_id IN [user_groups] — applied as a WHERE clause alongside whatever attribute predicates the query requires:

metadata_retrieval.py — exact-match query with ACL filter
async def search_metadata_store(
    query_text: str,
    user_groups: list[str],
    metadata_store,
    filters: dict = None,
    top_k: int = 20,
) -> list[dict]:
    """Keyword/attribute query on metadata store with group ACL filter."""

    # Build the WHERE clause — ACL filter is always included
    where = {
        "acl_group_id": {"$in": user_groups},  # non-negotiable
    }
    if filters:
        where.update(filters)  # e.g. {"category": "Active Contract", "workspace": "Legal"}

    results = await metadata_store.search(
        query=query_text,
        where=where,
        k=top_k,
    )
    return results

# Both paths run, then merge
async def hybrid_retrieve_with_acl(query, query_embedding, user_groups, ...):
    semantic_hits = await retrieve_with_acl(query_embedding, user_groups, ...)
    keyword_hits  = await search_metadata_store(query, user_groups, ...)
    merged = reciprocal_rank_fusion([semantic_hits, keyword_hits])
    return await verify_top_k(merged, user_ticket, otcs_client)

The metadata store path is where exact-match queries live, but it's also the path that most pipelines forget to protect. If you add the group filter to your vector store and nothing else, a user can still get unfiltered results from the keyword path. Defense in depth means both paths carry the same ACL key and both enforce it.

Caching Group Memberships

Group memberships change far less often than document content — usually only when HR updates a user's role or when a new project workspace is created. Cache the user-to-groups mapping per session with a TTL of 15–30 minutes as a safety net. But for real-time correctness, invalidate on permission events: when a user is added to or removed from an OpenText group, invalidate that user's cache entry immediately. The TTL catches everything else.

group_resolver.py — cached group lookup
class CachedGroupResolver:
    def __init__(self, otcs_base_url: str, cache, ttl: int = 1800):
        self.otcs_url = otcs_base_url
        self.cache = cache
        self.ttl = ttl  # 30 minutes default

    async def get_groups(self, ticket: str) -> list[str]:
        key = f"groups:{ticket[:16]}"
        cached = self.cache.get(key)
        if cached:
            return cached

        # OTCS REST: GET /api/v2/members/me/groups returns the user's group list
        groups = await self._fetch_groups_from_otcs(ticket)
        group_ids = [g["id"] for g in groups]
        self.cache.set(key, group_ids, ttl=self.ttl)
        return group_ids

Group sets are small — a few dozen integers, a few kilobytes at most. They're trivial to cache and trivial to invalidate by user ID. This is the opposite of caching a per-user document list, which can be megabytes, changes constantly as documents are created and reclassified, and can't be granularly invalidated. The cache key doesn't even need to carry information about what documents the user can see — just who the user is.

A Note on Orchestration Frameworks

Generic RAG orchestration libraries abstract the retrieval step in ways that can hide the filter clause from you. If you're using a high-level pipeline wrapper, verify that you can pass arbitrary metadata filters to the vector store call — specifically a multi-value acl_group_id IN [...] filter.

If you can't, build retrieval directly. ACL enforcement is not a feature to delegate to a framework abstraction. You need full control over what's passed to the vector store at query time, and you need the ability to add Stage 2 per-document verification without the framework intercepting the results.

⚠

Test with your highest-access users first. VPs, records managers, auditors — users with access to large swaths of the repository. They're the ones who expose naive ACL designs. If a broad-access user makes your filter clause time out or your memory spike, your design isn't production-ready regardless of how well it works for restricted users.

The Asymmetry Is the Design

The reason group-based filtering scales is simple: filter cardinality stays tiny (5–50 group IDs) while the protected document space stays huge (millions of nodes). You're not filtering more efficiently — you're filtering on the right dimension entirely.

Hybrid search adds a second reason to get this right. Two retrieval paths run in parallel. If only one carries the ACL filter, the other path is a permission leak — and Reciprocal Rank Fusion will happily surface those unprotected results at the top of the merged list. Defense in depth means acl_group_id in both stores, enforced independently on both paths, merged only after both have returned safe results.

OpenText was designed around groups for exactly this reason — groups are the stable, manageable unit of access control. The architecture should match the permission model.

Store the group on the chunk — in both stores. Cache the user's groups per session. Filter on groups on both retrieval paths. Re-verify on the top-k after fusion. That's the pattern that works at scale.
