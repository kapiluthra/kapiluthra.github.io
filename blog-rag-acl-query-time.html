ACL Enforcement: Index Time vs Query Time in OpenText RAG

The choice of when to apply permissions is not a security detail — it's an architecture decision that determines whether your system silently fails for restricted users or works reliably for everyone. Late enforcement sounds simpler until it hits production.

Kapil Uthra May 2026 ~11 min read
rag acl security opentext permissions

You indexed a million documents. Your retrieval is fast. Your LLM answers are good. Then you test with a restricted user — someone from finance, with access to a narrow slice of the repository — and they run a query. The system spends 900ms retrieving the top-20 results, applies your post-retrieval ACL check, and discards 17 of them because the user can't see them. Three results survive. Two are marginally relevant.

The user gets a weak answer. They ask the question again phrased differently. Same result. They stop using the system.

You just built a RAG system that works perfectly for administrators and fails for the users who matter most. The cause is where you chose to enforce permissions.

The Two Enforcement Moments

In any RAG pipeline over a permission-controlled content repository, you have two opportunities to apply access controls:

  • Early enforcement — before or during vector search. You constrain the search space to only documents the current user can access. The retrieval returns top-k results from the accessible set.
  • Late enforcement — after vector search and ranking. You retrieve the top-k from the full corpus, then filter out anything the user can't see. The LLM gets whatever survives.

Late enforcement is tempting because it feels simpler: index everything once, enforce permissions at the last moment. In practice, it systematically fails restricted users and introduces security risks that are subtle but real.

Why Late Enforcement Fails in Production

The Empty Results Trap

When you retrieve top-k from the full corpus and then filter, the number of results that survive depends on how much of the corpus the user can access. For a general user with broad permissions, late filtering works fine — most of their top-k results pass the check. For a restricted user, it's a different story.

Consider a finance analyst who can access 8,000 documents out of a 200,000-document corpus (4%). If your vector search retrieves the top-20 results from the full corpus, statistically about 0–1 of those results fall within their accessible set. You return an effectively empty context. The LLM hallucinates or says "I don't have information on that." The user is baffled — the document they're looking for is right there in the system.

The Latency Waste

Every document that gets filtered post-retrieval represents wasted work: a vector distance computation, potentially an OTCS REST fetch, a re-ranking pass. For a restricted user on a large corpus, you may be doing 10–20× more work than necessary — and delivering a worse result.

The Semantic Leakage Risk

This is the less obvious danger. Even after you filter out inaccessible chunks from the final context, those chunks influenced your retrieval scoring. A highly restricted document that ranked first in semantic similarity pulled other documents up in the ranking — documents that may be accessible but only tangentially relevant. The restricted document's semantic fingerprint is baked into your result ordering. As covered in detail in ACL-Aware RAG, post-retrieval filtering is not a clean security boundary.

⚠

Post-retrieval filtering is not a security control. It's a convenience layer. Re-ranking algorithms compute scores based on the full retrieved set before you filter — inaccessible content shapes the ranking even when removed from the context. For genuine access control, enforcement must happen before retrieval.

Early Enforcement: The Right Architecture

Early enforcement means constraining your vector store query to only search within documents the user is permitted to access. The user's accessible document set becomes a mandatory filter applied before any similarity computation.

The mechanism: at index time, every chunk is stored in the vector store with its OTCS node ID as a metadata field. At query time, you resolve the current user's accessible node IDs first, then pass that set as a filter constraint on the vector query. The vector store only computes similarity over the accessible subset.

The ACL Lookup Problem — and How Caching Solves It

The objection to early enforcement is always the same: "Resolving the user's accessible documents before every query is expensive. A user might have access to 50,000 nodes — I can't check each one per query."

This is a valid concern with a straightforward answer: cache the ACL resolution, not the retrieval.

OTCS permissions don't change per query — they change per permission event (role change, document reclassification, workspace restructure). For most enterprise users, their permission set is stable for hours or days. A cache with a 10–15 minute TTL eliminates the N+1 lookup problem entirely.

acl_resolver.py — cached ACL lookup with scoped indexing
import time, redis, json
from dataclasses import dataclass

@dataclass
class CachedACLResolver:
    otcs_base_url: str
    redis_client: redis.Redis
    ttl_seconds: int = 600  # 10 minutes

    def get_accessible_nodes(self, ticket: str, indexed_node_ids: set[int]) -> set[int]:
        # Try cache first
        cache_key = f"acl:{ticket[:16]}"  # partial ticket as key prefix
        cached = self.redis_client.get(cache_key)
        if cached:
            all_accessible = set(json.loads(cached))
            return all_accessible & indexed_node_ids  # intersect with indexed set

        # Cache miss — resolve from OTCS
        accessible = self._resolve_from_otcs(ticket, indexed_node_ids)
        self.redis_client.setex(
            cache_key,
            self.ttl_seconds,
            json.dumps(list(accessible))
        )
        return accessible

    def _resolve_from_otcs(self, ticket: str, node_ids: set[int]) -> set[int]:
        # Use OTCS search API with OTCSTicket — only returns accessible nodes
        # POST /api/v2/search with node_id filter: OTCS enforces ACL server-side
        accessible = set()
        batch = list(node_ids)
        for i in range(0, len(batch), 100):  # batch in groups of 100
            chunk = batch[i:i+100]
            ids_filter = " OR ".join(f"OTNodeID:{nid}" for nid in chunk)
            resp = self._search(ticket, ids_filter)
            accessible.update(r["id"] for r in resp.get("results", []))
        return accessible

The key insight in the implementation above: the OTCS search API itself enforces ACL. A search request authenticated with an OTCSTicket only returns nodes the ticket-holder can see. You're not writing custom permission logic — you're delegating to OTCS's existing access control and caching the result.

Index Time vs Query Time: When Does Each Apply?

Early enforcement at query time (resolving per user, per query) handles the common case. But there's a separate question about what to do at index time — specifically, whether to create user-specific indexes or a single shared index.

Shared Index (Recommended)

Index all documents into a single vector store with node ID metadata. Resolve user permissions at query time, pass as a filter. This is the right default: one index to maintain, permissions enforced dynamically, no synchronisation problem when permissions change.

User-Scoped Index (Avoid)

Create separate vector stores per user or user group. Index only documents each group can access. This sounds appealing because there's no runtime ACL resolution, but the maintenance cost is severe: when permissions change, you must re-index affected documents for every user scope that's impacted. In an enterprise with hundreds of user groups and frequently updated permissions, this becomes unmanageable quickly.

✓

Use group-level scoping for large deployments. Instead of per-user ACL resolution, resolve the user's effective OTCS groups (typically 5–20 groups per user) and tag embeddings with a list of group IDs that can access them. Querying then filters by group membership — fewer cache entries, faster resolution, and group permission changes only invalidate that group's cache entry.

Evaluating ACL Enforcement Quality

ACL enforcement is one of the few areas where you absolutely must run evaluation as multiple personas. A single test user with broad access will never surface the failure modes.

The test matrix I run on every deployment:

acl_test_matrix.py — multi-persona retrieval evaluation
from dataclasses import dataclass
from typing import Callable

@dataclass
class PersonaTest:
    name: str
    ticket: str          # OTCS ticket for this persona
    accessible_pct: float  # expected % of corpus accessible
    min_precision: float   # minimum acceptable precision@k

TEST_PERSONAS = [
    PersonaTest("admin",       ADMIN_TICKET,    1.00, 0.80),
    PersonaTest("finance",     FINANCE_TICKET,  0.15, 0.70),
    PersonaTest("contractor",  CONTRACTOR_TICKET, 0.04, 0.65),
    PersonaTest("read_only",  READONLY_TICKET, 0.60, 0.75),
]

def run_acl_eval(pipeline, test_queries: list[str], personas: list[PersonaTest]):
    for persona in personas:
        precisions = []
        for query in test_queries:
            results = pipeline.retrieve(query, ticket=persona.ticket)
            # Verify every result is accessible to this persona
            violations = [r for r in results if not can_access(persona.ticket, r.node_id)]
            assert len(violations) == 0, f"ACL violation for {persona.name}: {violations}"
            precisions.append(precision_at_k(results, k=5))

        avg_precision = sum(precisions) / len(precisions)
        assert avg_precision >= persona.min_precision, (
            f"{persona.name} precision {avg_precision:.2f} < threshold {persona.min_precision}"
        )

The critical assertion is the violation check: every retrieved result must be accessible to the querying persona. If any result fails this check, your pre-retrieval filter is not working correctly. This is a hard failure, not a metric to optimise — either your ACL enforcement is correct or it isn't.

The precision check catches the silent failure mode: a restricted user gets technically correct results (all accessible) but far fewer of them, degrading precision below an acceptable threshold. If your restricted personas show precision drops greater than 20% compared to broad-access users on the same queries, your accessible document set is too small — likely because your indexed corpus doesn't include the right documents for those personas' query patterns.

Handling Complex ACL Patterns in OTCS

OTCS permissions are more nuanced than a simple allowed/denied flag. A few patterns that require special handling:

  • Inherited vs overridden permissions: A workspace may grant broad read access, but a document inside it may have restricted permissions set directly on the node. Always resolve at the node level, never assume workspace inheritance is complete.
  • Time-limited access: Records Management workflows can grant temporary access (e.g., during an audit window). These need a shorter cache TTL — or event-driven invalidation triggered by the RM workflow state change.
  • External user federation: OTDS-federated external users may have identities from a different directory. Their OTCSTicket is valid but their group memberships resolve against the federated store. Confirm your ACL resolver handles federated group IDs correctly — the OTCS search API will enforce this, but your cache key must include the full identity context, not just the ticket prefix.
ℹ

Tag embeddings with access group ID instead of individual ACLs. If your OTCS ACL model is primarily group-based (most enterprise deployments are), store a list of group IDs with each embedding rather than a list of user IDs. This makes the filter clause compact, keeps the cache manageable, and means permission changes only invalidate the affected group's cache entry — not every individual user's cache.

The Foundational Difference

The difference between index-time, query-time early, and query-time late enforcement is not a detail to revisit once you're in production. It shapes whether your system works for the people who matter most — the restricted, role-scoped users who make up the majority of enterprise headcount.

Late enforcement is a prototype pattern that ships as a production system because it worked in testing with admin credentials. Early enforcement with a cached ACL resolver is a small upfront investment that guarantees every retrieved result is actually useful to the person who asked.

Early enforcement feels like overhead until you hit production — then it's the only thing standing between you and silent failures for 80% of your users.
