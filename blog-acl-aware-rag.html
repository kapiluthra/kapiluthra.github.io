ACL-Aware RAG: Why Your Enterprise AI Must Respect Document Permissions

Naive RAG ignores access controls. In enterprise, that's not a footnote — it's the reason most AI projects get blocked by security before they reach production. Here's the architecture that fixes it.

Every RAG tutorial does the same thing: ingest your documents, embed them, store them in a vector database, retrieve the top-k, generate an answer. It works beautifully in demos. Then you try to deploy it in an enterprise and the security team asks a simple question: if a user who can't see a document asks the AI about it, will the AI tell them?

In most naive RAG implementations, the answer is yes. The vector store has no concept of access control. It returns the most semantically similar chunks regardless of who's asking. Your AI will cheerfully summarise the CEO's salary review, the M&A target list, or the under-NDA client proposal — to anyone who phrases the right query.

This is the #1 reason enterprise AI projects get blocked by information security teams. And it's entirely solvable, if you build the right architecture from the start.

How OpenText Content Server ACLs Work

Before you can enforce permissions in a RAG system, you need to understand what permissions actually exist in the source system. OpenText Content Server has a mature, node-level permission model that most implementors only partially understand.

Every object in Content Server — document, folder, workspace, compound document — has an Access Control List composed of individual Access Control Entries. Each entry grants a specific permission set to a user or group:

  • See — the object appears in search results and folder listings
  • See Contents — required to read the document body
  • Modify — can edit metadata and content
  • Delete — can remove the object
  • Reserve / Unreserve — for checkout workflows

For RAG purposes, the critical permission is See Contents. A user might be able to see a document exists (it shows in search) but not read it. Your retrieval system needs to enforce at minimum the See Contents check before including any chunk in a response context.

Permissions also inherit through folder hierarchies by default, but can be overridden at any node. A folder might grant broad read access while a specific document inside it has restricted permissions. You cannot safely assume that folder-level ACLs tell you everything about document-level access.

⚠

The inheritance trap: I've seen implementations that check only the parent workspace ACL and assume child documents inherit identically. They don't, especially after document-level permission overrides from legal or HR workflows. Always check node-level ACLs.

The Two Approaches — and Why One Is Dangerous

When engineers first encounter the ACL problem in RAG, they typically reach for the obvious solution: retrieve the chunks as normal, then filter out any that the user shouldn't see before generating the answer. This is called post-retrieval filtering and it sounds reasonable. It isn't.

Post-Retrieval Filtering (Don't Do This)

The problem with filtering after retrieval is subtle but serious: re-ranking and context construction happen before you filter. Modern RAG pipelines use the retrieved chunks to compute relevance scores, build the LLM context window, and sometimes use early chunks to influence later retrieval steps. By the time you remove the forbidden chunks, the model has already "seen" them in the scoring phase.

More concretely: if chunk A (which the user can't see) scores highest in your similarity search, it influences which other chunks are selected, how the context is ordered, and potentially how the LLM frames its answer. You remove chunk A from the final context, but its semantic fingerprint is already baked into the retrieval result. The model may still echo its content through the other chunks it selected because of A's influence.

This is not theoretical. I've tested it on real corpora and seen answers contain paraphrased content from filtered chunks with enough consistency to be a genuine leak.

Pre-Retrieval Filtering (The Right Architecture)

The correct approach is to filter before retrieval — specifically, to only retrieve from a candidate set that the current user is authorised to see. This means your vector store query must be constrained to document IDs the user can access.

The key insight is that the OTCSTicket — the session token issued by Content Server authentication — already represents the user's identity and implicitly carries their permission context. Every OTCS REST API call made with that ticket will only return content the user is authorised to see. This is your enforcement mechanism.

Implementation: The Permission Resolver

The Permission Resolver sits between the user's query and the vector store. Its job is to produce a list of node IDs that the current user can access with See Contents permission. That list becomes a mandatory filter on every vector store query.

ℹ

The resolver does not need to enumerate every single document in the repository. It works on the set of documents that have been indexed into your vector store. You're intersecting the indexed set with the allowed set — not re-crawling OTCS on every query.

permission_resolver.py — build allowed node set for a user
import requests
from functools import lru_cache
from typing import set

class PermissionResolver:
    def __init__(self, otcs_base_url: str, indexed_node_ids: set[int]):
        self.base_url = otcs_base_url
        # The full set of nodes in our vector store
        self.indexed_nodes = indexed_node_ids

    def get_allowed_nodes(self, ticket: str) -> set[int]:
        """Return the subset of indexed nodes this ticket can see."""
        allowed = set()
        for node_id in self.indexed_nodes:
            if self._can_see_contents(ticket, node_id):
                allowed.add(node_id)
        return allowed

    def _can_see_contents(self, ticket: str, node_id: int) -> bool:
        # OTCS REST: GET /nodes/{id} returns 403 if no See permission
        # GET /nodes/{id}/content returns 403 if no See Contents
        url = f"{self.base_url}/api/v1/nodes/{node_id}/content"
        headers = {"OTCSTicket": ticket}
        resp = requests.head(url, headers=headers, timeout=5)
        return resp.status_code == 200

The naive version above checks every indexed node on every query — this doesn't scale. For a corpus of thousands of documents and concurrent users, you need to add two layers of caching.

Caching Strategy

Permission checks are expensive (one HTTP round-trip per document). But permissions don't change on every query. The right approach is a two-tier cache:

  • User-level cache: Cache the allowed node set per OTCSTicket for 5–10 minutes. Most users run multiple queries in a session; their permissions won't change mid-session.
  • Node-level cache: Cache the result of individual node permission checks for 15–30 minutes. Useful when the same document is checked by multiple users.
Cached permission resolver with TTL
import time
from dataclasses import dataclass, field

@dataclass
class CachedPermissionResolver:
    base_url: str
    indexed_nodes: set[int]
    user_cache_ttl: int = 300   # 5 minutes
    node_cache_ttl: int = 900   # 15 minutes

    _user_cache: dict = field(default_factory=dict)
    _node_cache: dict = field(default_factory=dict)

    def get_allowed_nodes(self, ticket: str) -> set[int]:
        now = time.time()
        if ticket in self._user_cache:
            result, ts = self._user_cache[ticket]
            if now - ts < self.user_cache_ttl:
                return result  # cache hit

        allowed = {
            nid for nid in self.indexed_nodes
            if self._check_node(ticket, nid, now)
        }
        self._user_cache[ticket] = (allowed, now)
        return allowed

    def _check_node(self, ticket: str, node_id: int, now: float) -> bool:
        key = (ticket, node_id)
        if key in self._node_cache:
            result, ts = self._node_cache[key]
            if now - ts < self.node_cache_ttl:
                return result
        result = self._can_see_contents(ticket, node_id)
        self._node_cache[key] = (result, now)
        return result

Wiring It Into Your Retrieval Pipeline

Once you have the allowed node set, you pass it as a filter to your vector store query. Most vector stores support metadata filtering — you stored the OTCS node ID alongside each chunk at index time, and now you use it as a retrieval filter.

ACL-filtered retrieval — vector store query with node filter
def retrieve_with_acl(
    query: str,
    ticket: str,
    vector_store,
    permission_resolver: CachedPermissionResolver,
    top_k: int = 10
) -> list[dict]:

    # Step 1: resolve allowed nodes for this user
    allowed_ids = permission_resolver.get_allowed_nodes(ticket)

    if not allowed_ids:
        return []  # user has no access to any indexed document

    # Step 2: query vector store with hard filter on node_id
    results = vector_store.similarity_search(
        query=query,
        k=top_k,
        filter={"node_id": {"$in": list(allowed_ids)}}
    )

    return results

The $in filter syntax varies by vector store. ChromaDB uses a where clause, Pinecone uses a metadata filter dict, Qdrant uses a Filter object. The principle is the same: the vector index only searches within the allowed document set.

Testing ACL Enforcement

ACL enforcement needs to be tested actively — not assumed to work because the code looks right. The test I run on every pipeline is what I call the poison document test:

  1. Index a document with clearly unique content (e.g., "Project AURORA budget is £14.7M")
  2. Restrict that document's See Contents to a single test user account
  3. Query the RAG system as a different user: "What is the budget for Project AURORA?"
  4. Verify the answer contains no reference to the specific figure or project name
  5. Query as the permitted user: verify the answer correctly retrieves the information
✓

Test with re-ranking enabled: If your pipeline uses a re-ranker, run the poison document test with it active. Re-rankers can reorder chunks in ways that expose information through the final answer even when the source chunk is filtered — especially if you're doing any form of multi-step retrieval.

The Harder Problem: Group Memberships and Dynamic ACLs

The implementation above works well when permissions are relatively stable. It becomes more complex when:

  • Users belong to many groups and group memberships change frequently (HR onboarding/offboarding)
  • Workspace-inherited permissions change at the folder level, invalidating cached node checks
  • Time-limited access is granted (e.g., via Records Management workflows)
  • External users connect through OTDS federation with a different identity store

For these cases, you need an event-driven cache invalidation strategy. OTCS emits audit events for permission changes — you can subscribe to these via the Event API or a message queue integration and use them to selectively purge your permission cache entries. This is more complex but essential for systems where permissions change frequently.

The simpler fallback: keep your cache TTL short (under 5 minutes) and accept the performance trade-off. For most enterprise RAG use cases, a 5-minute window is acceptable — permission changes don't take effect instantaneously in OTCS either.

Why This Is the #1 Enterprise AI Blocker

I've sat in more than a dozen conversations with information security teams about enterprise AI deployments. In almost every case, the first question is some variant of: "How do you ensure the AI only answers based on what the user is authorised to see?"

When the answer is "we filter after retrieval" or "we only index public documents," the project stalls. The security team is right to push back. Post-retrieval filtering is insufficient. "Only public documents" defeats the entire value proposition — the valuable content is the restricted content.

When the answer is "we resolve permissions against the source system's ACL before retrieval, using the same ticket that gates direct document access" — that's an answer security teams can work with. It's auditable, it's consistent with existing controls, and it doesn't require a separate authorisation policy to maintain.

ACL-aware RAG isn't an advanced feature. It's the minimum viable security bar for any enterprise deployment. Build it in from the start — retrofitting it later means reindexing your entire corpus with permission metadata and rewriting your retrieval layer.
