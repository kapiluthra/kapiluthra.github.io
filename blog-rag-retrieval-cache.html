Why You Need a Retrieval Cache in Your OpenText RAG Pipeline

The same question gets asked every day. Your pipeline re-embeds, re-searches, and re-fetches from OTCS every time. A retrieval cache makes that 45ms instead of 2.4 seconds — and in production, that difference is between a system people use and one they abandon.

Kapil Uthra May 2026 ~10 min read
rag caching performance opentext redis

"What's the Q3 budget policy?" Someone asks it at 9 AM. Someone else asks it at 2 PM. A third person asks it the next morning after a team meeting. Each time, your RAG pipeline computes embeddings, runs a vector search, issues REST calls to OpenText Content Management, re-ranks results, and generates an answer. 2.4 seconds. Three times. Same answer.

A retrieval cache would have served the second and third response in 45ms from memory. Over a month, as your cache hit rate climbs to 60–80% on a real enterprise corpus, you've cut retrieval latency by 50× for the majority of queries. The cost drop is proportional. The user experience is night and day.

This isn't premature optimisation. Caching is the difference between a prototype and a production system. And the enterprise ECM context makes it even more valuable than in generic RAG deployments — because OTCS query patterns are bursty and deeply repetitive by department.

Why OpenText RAG is Especially Cache-Friendly

Generic RAG deployments on diverse corpora see moderate repetition — maybe 15–25% of queries overlap within a day. Enterprise ECM deployments on OpenText are different. The query population is structured around business roles and repeating work cycles:

  • A legal team asks about the same contract templates every time a new deal starts
  • An operations team asks about the same policy documents before each audit
  • Finance asks the same budget and reporting questions every month-end
  • HR asks the same onboarding document questions every new joiner wave

In production OTCS RAG deployments, 30–50% of queries repeat within a 4-hour window from users in the same department. The same semantic question, phrased slightly differently ("show me the Q3 policy" vs "what does the Q3 policy say" vs "Q3 budget rules") but pointing at the same retrieval result set.

Your pipeline treats every one of these as a novel query. It shouldn't.

The Latency Breakdown

Before building a cache, it helps to understand where the time actually goes in a typical OTCS RAG query. In a production deployment I instrumented, a single end-to-end query broke down like this:

Latency profile — single RAG query without cache
# Measured on a 50k-document OTCS corpus, AWS eu-west-1

Embed query (OpenAI ada-002)        ~120ms
Vector store search (Qdrant, k=20)  ~180ms
OTCS REST fetch (10 nodes, serial)  ~900ms   # dominant cost
Re-rank (cross-encoder)             ~200ms
LLM generation (GPT-4o, 600 tok)   ~900ms
─────────────────────────────────────────
Total                              ~2,300ms

# With cache hit on retrieval layer:
Cache lookup (Redis GET)             ~5ms
LLM generation (GPT-4o, 600 tok)   ~900ms
─────────────────────────────────────────
Total                               ~905ms   # 2.5× faster end-to-end

# With cache hit on full response:
Cache lookup (Redis GET)             ~5ms
─────────────────────────────────────────
Total                                 ~5ms   # 460× faster

The OTCS REST fetch is the dominant cost — nearly 900ms just to retrieve the content of 10 documents over the network. This is the layer that benefits most from caching. Even partial caching of the retrieval result set (skipping vector search and OTCS fetch, but still running LLM generation) cuts latency from 2.3 seconds to under 1 second.

What to Cache — and What Not To

There are three distinct layers you can cache, each with different trade-offs:

Layer 1: Retrieval Results (Recommended)

Cache the output of your retrieval pipeline: the ranked list of chunks (with node IDs, metadata, and text) that would be passed to the LLM. The cache key is a hash of the query embedding plus any active filters.

This is the right default. You skip the expensive vector search and OTCS fetch, but still run LLM generation fresh — which means your answers stay up to date even if the cached retrieval is slightly stale. The risk surface is small.

Layer 2: Full Response (Use Sparingly)

Cache the complete LLM-generated answer for an exact query. Fastest possible hit — 5ms total. But risky: if the underlying documents change, users get cached answers that are factually outdated with no indication of staleness. Reserve this for truly static content (regulatory text, archived policies) where the underlying documents don't change.

Layer 3: OTCS Node Content (Always Worth It)

Cache the raw content fetched from OTCS for individual node IDs, keyed by (node_id, version_number). This is safe to cache aggressively because the key is version-scoped — a new document version gets a new key automatically. It eliminates the per-query OTCS REST round-trips even when the retrieval result set itself isn't cached.

⚠

Don't cache by document ID alone. Cache (node_id, version_number). If someone updates a document in OTCS, the version number increments and the old cache entry becomes unreachable — automatic invalidation without a complex event system.

The Cache Key Design

Getting the cache key right is the most important design decision. A bad key causes either cache misses (you never hit) or cache collisions (different queries return the same result).

For retrieval result caching, the key needs to capture everything that would change the retrieval output:

cache_key.py — building a stable retrieval cache key
import hashlib, json
import numpy as np

def make_retrieval_cache_key(
    query_embedding: list[float],
    filters: dict,
    top_k: int,
    max_doc_date: str # ISO date of most recently modified indexed doc
) -> str:
    # Quantise embedding to 2 decimal places — groups semantically
    # similar queries into the same bucket (semantic hashing)
    quantised = [round(x, 2) for x in query_embedding]

    payload = {
        "emb": quantised,
        "filters": filters,
        "k": top_k,
        "max_date": max_doc_date,
    }
    raw = json.dumps(payload, sort_keys=True)
    return "rag:retrieval:" + hashlib.sha256(raw.encode()).hexdigest()[:32]

The quantisation step is the key insight. By rounding the embedding to 2 decimal places before hashing, semantically near-identical queries — "Q3 budget policy", "what is the Q3 budget policy", "tell me about Q3 budget" — often produce embeddings that round to the same quantised vector, hitting the same cache entry. This is semantic hashing: you get cache hits on paraphrase queries, not just exact string matches.

The max_doc_date component ensures that when new documents are indexed into OTCS, the cache key shifts — old cached results for that query will miss and get recomputed with the fresh corpus. No explicit invalidation needed for content updates.

Implementation: Start Simple, Graduate to Redis

The right progression for a retrieval cache is not to design the full distributed solution from day one. Start with an in-process dict and TTL. Graduate when you outgrow it.

Stage 1: In-Process Dict (Under 10 RPS)

retrieval_cache.py — in-process cache with TTL
import time, json
from dataclasses import dataclass, field

@dataclass
class RetrievalCache:
    ttl_seconds: int = 900          # 15 minutes default
    max_entries: int = 500
    _store: dict = field(default_factory=dict)
    _hits: int = 0
    _misses: int = 0

    def get(self, key: str) -> list | None:
        entry = self._store.get(key)
        if entry is None:
            self._misses += 1
            return None
        chunks, ts = entry
        if time.time() - ts > self.ttl_seconds:
            del self._store[key]
            self._misses += 1
            return None
        self._hits += 1
        return chunks

    def set(self, key: str, chunks: list):
        if len(self._store) >= self.max_entries:
            # Evict oldest entry
            oldest = min(self._store, key=lambda k: self._store[k][1])
            del self._store[oldest]
        self._store[key] = (chunks, time.time())

    def hit_rate(self) -> float:
        total = self._hits + self._misses
        return self._hits / total if total > 0 else 0.0

Stage 2: Redis (10+ RPS or Multiple Workers)

Once you have multiple worker processes or need the cache to survive restarts, move to Redis. The interface stays identical — only the backend changes.

redis_cache.py — drop-in Redis backend
import redis, json, zlib

class RedisRetrievalCache:
    def __init__(self, url: str = "redis://localhost:6379", ttl: int = 900):
        self.r = redis.from_url(url, decode_responses=False)
        self.ttl = ttl

    def get(self, key: str) -> list | None:
        raw = self.r.get(key)
        if raw is None:
            return None
        return json.loads(zlib.decompress(raw))

    def set(self, key: str, chunks: list):
        compressed = zlib.compress(json.dumps(chunks).encode())
        self.r.setex(key, self.ttl, compressed)  # TTL in seconds

The zlib compression is worth adding from the start — chunk payloads are text-heavy and compress well (typically 60–70% reduction). Redis memory is not free, especially when you're caching hundreds of thousands of chunk sets.

Cache Invalidation Without an Event System

The hardest part of caching isn't the cache — it's knowing when to invalidate it. Full event-driven invalidation (subscribing to OTCS change events, purging specific keys) is complex to build and maintain. For most deployments, TTL-based expiry plus content-date keying is sufficient.

The pattern:

  • Your incremental sync job runs on a schedule (hourly, or nightly for low-change corpora)
  • After each sync, record the max_modify_date of all indexed documents
  • Include this date in every cache key
  • When documents update and the sync runs, the max_modify_date shifts — all cache keys for affected queries automatically miss and recompute
✓

Track hit rate by department or user role. Legal queries may have an 80% hit rate (highly repetitive). Strategy queries from senior leadership may have 10% (diverse, unpredictable). High variance between cohorts tells you which teams are underserved — and may indicate that their query templates could be standardised to improve cache efficiency.

Wiring the Cache Into Your Pipeline

The cache wraps the retrieval step. The LLM generation step always runs on a cache hit — you're caching the retrieved context, not the generated answer.

cached_rag.py — cache-aware retrieval pipeline
async def retrieve_with_cache(
    query: str,
    filters: dict,
    embedder,
    vector_store,
    otcs_fetcher,
    cache: RedisRetrievalCache,
    max_doc_date: str,
    top_k: int = 10,
) -> list[dict]:

    # 1. Embed query
    embedding = await embedder.embed(query)

    # 2. Build cache key and check
    key = make_retrieval_cache_key(embedding, filters, top_k, max_doc_date)
    cached = cache.get(key)
    if cached is not None:
        return cached  # ~5ms path

    # 3. Cache miss — full retrieval
    node_ids = await vector_store.search(embedding, filters, top_k)
    chunks = await otcs_fetcher.fetch_chunks(node_ids)

    # 4. Store and return
    cache.set(key, chunks)
    return chunks

Production Considerations

A few things that bite teams in production that aren't obvious in development:

  • Cache warming: On a cold start (new deployment, Redis flush), your first wave of queries all miss. If you have high traffic at startup, this can cause a latency spike. Pre-warm by replaying the top 50 most common queries from your logs before going live.
  • ACL interaction: If you're doing ACL-aware retrieval, the cache key must include the user's permission scope. Two users with different permissions asking the same question should not share a cache entry. Add a hashed representation of the user's access group to the key.
  • Cache stampede: When a popular cached entry expires, many concurrent requests may all miss simultaneously and trigger parallel retrievals. Solve with a simple mutex: the first request acquires a lock on the key while computing; others wait 200ms and retry.
ℹ

Start measuring hit rate from day one. Log every cache hit and miss with a timestamp and query hash. A hit rate below 20% after one week of real traffic means your semantic hashing quantisation is too fine — try rounding to 1 decimal place instead of 2. Above 80% may mean your TTL is too long and users are getting stale results.

Caching is Not Premature Optimisation

Teams consistently underestimate how repetitive enterprise query patterns are. When the same 50 questions account for 40% of your daily query volume, not caching them is the inefficiency — not caching them.

The implementation is straightforward. Start with a 20-line in-process dict. Add TTL. Measure the hit rate. Graduate to Redis when you need shared state across workers. The cache key design — embedding quantisation plus content date — handles 90% of real-world cache invalidation concerns without needing a complex event subscription system.

50 queries per second or 1 query every 10 seconds. The retrieval cache is what separates those two outcomes at scale.
