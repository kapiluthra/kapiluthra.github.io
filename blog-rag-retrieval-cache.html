Caching in OpenText RAG: Skip the Pipeline, Not Just the Fetch

The same question gets asked every day. Your RAG pipeline re-embeds, re-searches, and calls the LLM every time. The bottleneck isn't OTCS — it's the LLM. Cache the right layers and repeat queries go from 1500ms to 5ms.

Kapil Uthra May 2026 ~10 min read
rag caching performance opentext redis

"What's the Q3 budget policy?" Someone asks it at 9 AM. Someone else at 2 PM. A third person the next morning. Each time, your pipeline embeds the query, runs a vector search, and sends the retrieved chunks to an LLM for generation. ~1500ms. Three times. Same answer.

A properly layered cache returns the second and third response in under 10ms. Over a month on a real enterprise deployment — where 30–50% of queries repeat within hours — that's the difference between a system under constant load and one that scales gracefully. But to build the cache correctly, you first need an accurate model of where the time actually goes.

First: What Actually Happens at Query Time

A common misconception is that OpenText Content Management is in the query path — that each user query triggers a REST fetch from OTCS. It doesn't. OTCS is only accessed at index time.

The two phases are completely separate:

Indexing — runs once (or on schedule)

  1. Fetch document content from OTCS via REST API
  2. Extract text, chunk, embed
  3. Store chunks + embeddings + metadata (including node ID) in vector DB

After this, OTCS is out of the picture for query serving. The vector DB holds everything needed to answer questions.

Query time — runs on every user request

  1. Embed the user's query
  2. Vector search → top-k chunks (already stored from indexing)
  3. Optional: re-rank chunks
  4. Pass chunks as context to LLM → generate response
ℹ

OTCS does get called at query time in two specific cases: (1) fetching the source document for inline citation display — usually async after the answer is returned, and (2) re-validating ACLs to confirm the user still has access to the cached chunks. Both are deliberate, bounded calls — not the main query path.

The Real Latency Breakdown

Once you understand the correct query-time architecture, the bottleneck becomes obvious:

StageTypical latency
Query embedding (local / API)50–200ms
Vector DB search30–100ms
Re-ranking (cross-encoder)100–300ms
LLM generation ← bottleneck1000–3000ms

The LLM is responsible for 60–80% of total latency. Every other optimisation — faster embeddings, a better vector index, tighter chunking — is noise compared to the cost of an LLM call. Caching in RAG is primarily about avoiding redundant LLM calls, not about avoiding OTCS fetches.

The Three Cache Layers

There are three distinct places to intercept the pipeline, each with different latency savings, staleness risk, and implementation complexity.

Layer 1 — Full Response Cache (Greatest Impact)

Cache the complete LLM-generated answer keyed on a hash of the user's query text (after normalisation). On a hit, skip the entire pipeline — embedding, vector search, and LLM generation — and return the cached answer directly.

Latency saved: 1500–3500ms → ~5ms. This is the highest-value layer. The risk is staleness: if the underlying documents change, the cached answer may be outdated. Mitigate with a short TTL (15–30 minutes for live content, longer for stable policy documents) and include a "cached at" timestamp in the response so users know when the answer was generated.

Layer 2 — Embedding Cache (Low Effort, Moderate Savings)

Cache the embedding vector keyed on the exact query string. The same query asked twice doesn't call the embedding model a second time.

Latency saved: 50–200ms. Low staleness risk — the embedding for a query string doesn't change unless you swap embedding models. Simple to implement: a dict or Redis key of sha256(query_text) → float[]. Worth adding from day one; it's five lines of code.

Layer 3 — Retrieval Results Cache (Middle Ground)

Cache the ranked chunk list returned by vector search, keyed on the quantised embedding plus active filters. On a hit, skip vector search but still run LLM generation. The answer stays fresh (LLM re-generates each time) while the expensive similarity computation is skipped for repeated queries.

Latency saved: 130–400ms. This is the right layer to cache when you need answer freshness but want to reduce vector DB load. It's also the layer that benefits from semantic hashing: by rounding embedding dimensions to 1–2 decimal places before hashing, paraphrase queries ("Q3 budget policy" / "what is the Q3 policy" / "Q3 budget rules") map to the same cache entry.

Implementation

The cleanest structure is a wrapper that checks each cache layer in order before proceeding to the next pipeline stage.

rag_pipeline.py — three-layer cache-aware pipeline
import hashlib, json, time
from dataclasses import dataclass

@dataclass
class CachedRAGPipeline:
    embedder: object
    vector_store: object
    llm: object
    cache: object          # Redis or in-process dict
    response_ttl: int = 900   # 15 min for full responses
    retrieval_ttl: int = 1800  # 30 min for retrieval results

    async def query(self, user_query: str, filters: dict = None) -> dict:

        # ── Layer 1: full response cache ──────────────────────
        l1_key = "resp:" + _hash(_normalise(user_query))
        cached_resp = self.cache.get(l1_key)
        if cached_resp:
            return {**json.loads(cached_resp), "cache": "L1"}

        # ── Layer 2: embedding cache ───────────────────────────
        emb_key = "emb:" + _hash(user_query)
        cached_emb = self.cache.get(emb_key)
        if cached_emb:
            embedding = json.loads(cached_emb)
        else:
            embedding = await self.embedder.embed(user_query)
            self.cache.set(emb_key, json.dumps(embedding), ttl=86400)

        # ── Layer 3: retrieval results cache ──────────────────
        l3_key = "ret:" + _hash(_quantise(embedding), filters)
        cached_chunks = self.cache.get(l3_key)
        if cached_chunks:
            chunks = json.loads(cached_chunks)
        else:
            chunks = await self.vector_store.search(embedding, filters)
            self.cache.set(l3_key, json.dumps(chunks), ttl=self.retrieval_ttl)

        # ── LLM generation (always fresh on L2/L3 hits) ───────
        answer = await self.llm.generate(user_query, chunks)
        result = {"answer": answer, "chunks": chunks, "cache": "miss"}
        self.cache.set(l1_key, json.dumps(result), ttl=self.response_ttl)
        return result


def _normalise(query: str) -> str:
    # Lowercase, collapse whitespace — catches trivial paraphrases
    return " ".join(query.lower().split())

def _quantise(embedding: list[float], filters: dict = None) -> str:
    # Round to 2dp — bins semantically similar queries together
    q = [round(x, 2) for x in embedding]
    return json.dumps({"q": q, "f": filters or {}}, sort_keys=True)

def _hash(*parts) -> str:
    raw = "|".join(str(p) for p in parts)
    return hashlib.sha256(raw.encode()).hexdigest()[:32]

Invalidation Without an Event System

The question that always comes up: "If a document gets updated in OTCS, the cached answer is stale. How do you handle that?"

For most deployments, TTL-based expiry is sufficient. Policy documents don't change hourly. A 15-minute TTL means a user will see an outdated answer for at most 15 minutes after a content update — acceptable for almost every enterprise use case.

For content that changes more frequently, tie the L3 retrieval cache key to the max_modify_date of your indexed documents. Your incremental sync job (which runs on schedule) updates this value after each sync. When it shifts, all L3 keys for affected queries automatically miss on the next request and recompute against the fresh index. No explicit purge logic required.

⚠

ACL interaction with L1 cache: If two users with different permissions ask the same question, they must not share a full-response cache entry. Include a hashed representation of the user's access group in the L1 key. Users with identical permissions can share a cache entry; users with different permission scopes must not.

What to Measure

A cache without metrics is a black box. Log these from day one:

  • Hit rate per layer — L1, L2, L3 separately. A low L1 hit rate with a high L3 hit rate means users are getting fresh LLM answers for repeated retrieval patterns, which is often the right trade-off.
  • Hit rate per department or user role — legal queries repeat more than strategy queries. High variance between cohorts signals where to tune TTLs or invest in query template standardisation.
  • Latency distribution by cache outcome — P50 and P95 latency for L1 hits, L3 hits, and full misses. This is the number that makes the case for the cache to stakeholders.
Logging cache outcomes for observability
import logging, time

async def query_with_telemetry(pipeline, user_query, filters, user_role):
    t0 = time.perf_counter()
    result = await pipeline.query(user_query, filters)
    latency_ms = (time.perf_counter() - t0) * 1000

    logging.info("rag_query", extra={
        "cache_layer": result["cache"],     # "L1", "L2", "L3", or "miss"
        "latency_ms": round(latency_ms),
        "user_role": user_role,
        "query_hash": _hash(_normalise(user_query)),
    })
    return result

Caching is Not Premature Optimisation

Enterprise query patterns are bursty and repetitive by department. A legal team asks the same contract questions every deal cycle. Finance asks the same reporting questions every month-end. In production, a well-tuned L1 cache hitting at 50%+ cuts your LLM API costs in half and makes the system feel instant for the majority of real-world queries.

The implementation starts at 30 lines of Python and a Redis instance. Start with an in-process dict if Redis feels like overhead. Measure hit rates from the first week. The data will make the case for investment better than any architecture diagram.

The bottleneck is the LLM. Cache accordingly.
