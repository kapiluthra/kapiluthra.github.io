OTDS SSO Deep Dive: The Authentication Layer Most xECM Implementations Get Wrong

OTCSTicket vs OTDS federation, user partition pitfalls, the login-attribute mapping trap, and why co-locating OTDS with Content Server causes intermittent auth failures that look like permission problems but aren't.

Kapil Uthra June 2026 ~14 min read
otdsssoopentextauthenticationxecm

The most common class of support ticket in xECM implementations is some variation of: "user can log into SAP fine but gets a 401 in the document panel." I've debugged this across a dozen engagements in five countries. The root cause is almost never a permissions problem. It's almost always an identity federation failure in OTDS that produces an error that looks exactly like a permissions problem.

This post is the write-up I wish had existed when I started doing xECM work. It covers the OTDS architecture, the three misconfigurations that cause 90% of auth failures, and how to diagnose them systematically rather than by trial and error.

OTCSTicket vs OTDS: What's the Difference?

Content Server has two authentication mechanisms that often get confused:

OTCSTicket (native CS auth)
  • POST credentials to /api/v1/auth, receive a session ticket
  • Ticket passed in every subsequent API request header
  • Expires after configurable timeout (default 30 min)
  • User identity managed in Content Server's own user database
  • Used by: direct API clients, scripts, the RAG pipeline
OTDS federation (SSO)
  • OpenText Directory Services — a separate identity broker service
  • Federates external identity providers: Active Directory, SAML IdPs, SAP logon
  • Issues OTDS tokens that Content Server accepts for SSO
  • Enables "log into SAP, open document panel seamlessly"
  • Used by: xECM for SAP, xECM for Salesforce, web UI SSO

When an xECM for SAP user gets a 401 in the document panel, they are hitting the OTDS path — not the OTCSTicket path. The SAP system passes the user's identity to OTDS, OTDS maps it to a Content Server user, and Content Server checks that user's permissions. A failure at any point in the mapping chain produces a generic 401.

The Three Misconfigurations That Cause 90% of Failures

1. User Partition Name Mismatch

OTDS organises users into partitions. When Content Server is registered as a resource in OTDS, it's assigned to specific partition(s). The partition name must match exactly between the OTDS resource configuration and the partition where your federated users live.

The symptom: SSO works for some users but not others. Investigation usually reveals the working users were created directly in the default partition, while failing users were imported from Active Directory into a custom partition that isn't assigned to the CS resource.

Diagnostic — check which partitions are mapped to your CS resource
# OTDS REST API call to list resource partition assignments
GET /otdsws/rest/resources/{resource_id}/partitions
Authorization: OTDSTicket {admin_ticket}

# Response will show assigned partitions
# Compare against the partition containing your federated users
# They must match — a user in partition A cannot auth to a resource
# that only has partition B assigned

2. Login Attribute Mapping

This is the most subtle and the most common failure mode. OTDS needs to know which attribute in the external identity source (Active Directory, SAP) maps to the user's Content Server login name. This mapping is configured per partition and per identity provider.

The typical SAP xECM case: SAP sends the user's SAP logon name (e.g. KUTHRAK). OTDS needs to find a Content Server user with that login. If the CS user was created with a different convention (e.g. kapil.uthra@company.com or DOMAIN\kuthrak), the mapping fails silently and OTDS returns a 401.

⚠

Silent failure mode: OTDS does not return a useful error message when a login attribute mapping fails. The 401 looks identical to a genuine "user doesn't exist" or "wrong password" response. Always check the OTDS audit log, not the Content Server log, when diagnosing these failures.

Checking OTDS audit log for federation failures
# OTDS admin console → Audit → filter by event type "AUTHN_FAILURE"
# Or via REST API:
GET /otdsws/rest/audit?eventType=AUTHN_FAILURE&limit=50
Authorization: OTDSTicket {admin_ticket}

# Look for entries showing:
# - principal: the SAP logon name being passed
# - reason: "user not found" vs "invalid credentials"
# "user not found" = login attribute mapping issue
# "invalid credentials" = user exists but password/token wrong

3. Resource Partition Assignment (the "ghost user" problem)

A federated user can exist in OTDS — synchronised successfully from Active Directory, visible in the OTDS admin console — but still fail to authenticate to Content Server if the user's OTDS partition is not assigned to the Content Server resource.

This manifests as the "ghost user" problem: the user is visible in OTDS, has a valid session token, can access other OTDS-protected resources — but Content Server returns 401. The fix is straightforward (assign the partition to the CS resource) but the diagnosis is non-obvious because everything looks correct in the OTDS UI except the resource-partition assignment screen, which is several menus deep.

The Co-Location Performance Trap

OTDS and Content Server have fundamentally different resource profiles:

  • Content Server is I/O bound — database queries, file system access, full-text indexing
  • OTDS is CPU bound under load — token validation, LDAP sync, federation operations are CPU-intensive

When co-located on the same VM or container, they compete for CPU under load. The result is intermittent authentication failures that appear random — they're actually caused by OTDS token validation timing out when Content Server spikes CPU usage during indexing or heavy query periods.

✓

Standard recommendation since Orion: OTDS runs on a dedicated host (VM or container) with its own CPU allocation. This is not gold-plating — it's the minimum viable setup for a production xECM deployment. At Orion Finland, separating OTDS eliminated the intermittent 401 errors that had been attributed to a "known OTDS bug" for six months.

OTDS Sizing Guidelines

OTDS is consistently undersized in initial deployments because teams focus sizing effort on Content Server. The minimum I configure for a production xECM deployment:

  • CPU: 4 vCPUs dedicated (not shared). OTDS token operations are synchronous and CPU-bound — shared vCPUs cause variable latency that manifests as auth timeouts.
  • Memory: 8GB minimum. OTDS caches partition data aggressively — insufficient memory causes cache evictions under load that produce latency spikes.
  • Storage: SSD for the OTDS database. LDAP sync operations under large directories (50K+ users) are disk-intensive.
  • Network: OTDS must be able to reach both the LDAP/AD server and the Content Server with sub-10ms round trips. High-latency federation calls are the second most common cause of intermittent auth failures after the three misconfigurations above.

A Diagnostic Checklist

When "user can log in but gets 401 in document panel," work through this in order:

  • Check OTDS audit log for AUTHN_FAILURE events — is the user failing at OTDS or at Content Server?
  • If failing at OTDS: check login attribute mapping for the user's partition and identity provider
  • If failing at Content Server: check that the user's OTDS partition is assigned to the CS resource
  • If the failure is intermittent (not always 401): check OTDS CPU usage during failure windows — likely a co-location contention issue
  • If the failure is after N minutes of inactivity: the OTDS token TTL is shorter than the SAP session timeout — configure token refresh or increase TTL

I've never encountered an xECM 401 that wasn't one of these five cases. If you're hitting something that doesn't fit, the OTDS REST API for audit events is the most reliable diagnostic source — it captures what the admin console UI hides.
