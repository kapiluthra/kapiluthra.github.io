Node ID Preservation: The Link Back to OpenText That Most RAG Pipelines Break

Your RAG pipeline retrieved the right chunk. Now what? If you can't trace it back to the source document in Content Server, you've built a citation machine with broken links. Here's how to preserve the chain from chunk to node.

Kapil Uthra May 2026 ~9 min read
rag opentext node-id citations traceability

The retrieval worked. The chunk is correct. The LLM synthesises a clear, accurate answer and — in the citations — references "Annual EHS Report 2023, Section 4.2." The user reads the answer, finds it credible, and asks the natural follow-up question: "Can you show me the document?"

You have no link. The chunk exists in your vector store as floating text, severed from the Content Server node it came from. You can tell the user the document name. You cannot tell them where it lives, which version it is, or how to open it. If they're lucky, they'll find it through OTCS search. If they're not, they'll go looking and fail — and the next time the AI cites something, they won't trust it.

This is one of the most common and most preventable failures in enterprise RAG deployments. Node ID preservation is five lines of code to add at index time and months of user trust to rebuild if you skip it.

What the OTCS Node ID Is and Why It's Stable

Every object in OpenText Content Server — documents, folders, workspaces, compound documents, wiki pages — is assigned a DataID at creation time. This is an integer, typically in the millions for mature OTCS deployments (e.g., 4821993). It is the primary key of the DTreeCore database table and the universal identifier for that object throughout the OTCS system.

The DataID has a critical property: it never changes. Rename the document — same DataID. Move it to a different folder — same DataID. Check in a new version — same DataID. Copy the document — the copy gets a new DataID; the original keeps its own. Even if the document moves between workspaces or gets reclassified, the DataID is permanent for the lifetime of the object.

This permanence is what makes the DataID the right anchor for your linkback. It's not the document name (which changes), not the file path (OTCS doesn't have traditional paths), not the URL (which encodes folder navigation and can be reconstructed from the DataID). It's the DataID.

The direct link pattern for any OTCS document is:

OTCS direct link patterns — classic UI and REST API
# Classic Content Server UI deep link
https://{otcs-host}/otcs/cs.exe?func=ll&objType=1&objID={node_id}

# Smart UI deep link (OTCS 16.x+)
https://{otcs-host}/otcs/cs.exe/open/{node_id}

# REST API: get node metadata
GET /api/v2/nodes/{node_id}

# REST API: download content
GET /api/v2/nodes/{node_id}/content

# REST API: get version list
GET /api/v2/nodes/{node_id}/versions

All of these are stable as long as you have the DataID. The Classic UI link is the most universally compatible — it works across OTCS versions from 10.5 to current. The Smart UI link is cleaner for modern deployments. The REST API endpoints are what your pipeline uses to retrieve metadata and content programmatically.

The Four Places Node ID Gets Lost

Node ID loss is almost never intentional. It happens at the seams between pipeline stages — places where one component passes data to the next, and the handoff loses the metadata that wasn't explicitly included. Here are the four failure points I've seen repeatedly in real pipelines:

1. During Text Extraction

You call pdfplumber.open(file) or python-docx's Document(file). These libraries return text. They know nothing about where the file came from or what node it represents. If you extract text to a variable or write it to a file without explicitly carrying the node_id alongside it, the provenance is already gone.

The fix is trivially simple: never extract text in isolation. Always extract into a data structure that includes the node ID as a field.

2. During Chunking

Text splitters — LangChain's RecursiveCharacterTextSplitter, LlamaIndex's SentenceSplitter, or custom implementations — take text as input and return a list of text strings. If you pass raw text to the splitter, you get raw text chunks back. The node_id is not threaded through.

The fix: wrap your chunker to accept a source document object and propagate its metadata to each output chunk.

3. During Vector Store Ingestion

Most vector store client libraries have two ingestion patterns: pass a list of strings (no metadata), or pass a list of Document objects with a metadata dict. If you use the string-list path — which is what most quick-start examples demonstrate — the node_id has nowhere to go.

The fix: always use the document-with-metadata ingestion path, and always include node_id in the metadata dict.

4. During Retrieval

Vector store retrieval returns either a list of text strings or a list of document objects. Many retrieval wrapper functions are written to extract just the .page_content or .text field from returned documents — discarding the .metadata dict that contains node_id. The retrieval result passed to the LLM is a list of strings with no provenance.

The fix: return the full document object from your retrieval function, never just the text. Strip metadata only at the point of LLM context construction, and only strip what the LLM doesn't need — keep node_id in the citation layer.

⚠

The silent drop: None of these failures raise an exception. The pipeline runs, the LLM generates answers, the system looks correct in demos. The node_id is simply absent from the output. You only discover the problem when a user asks for the source document and you have nothing to show them.

The Preservation Pattern — End to End

The solution is to use a typed data structure that carries node_id through every pipeline stage, and to never accept raw strings at a stage boundary. Pass objects, not strings.

chunk.py — typed chunk dataclass, node_id as first-class field
from dataclasses import dataclass, field
from typing import Optional

@dataclass
class OTCSChunk:
    # Source provenance — never optional
    node_id: int
    node_name: str
    indexed_version: int

    # Position within the source document
    chunk_index: int
    total_chunks: int

    # Content — two forms
    text: str            # raw chunk text, for display
    embed_text: str      # header + text, used for embedding

    # Optional enrichment
    workspace_name: str = ""
    doc_class: str = ""
    status: str = ""
    modify_date: str = ""

    def chunk_id(self) -> str:
        """Stable unique ID for this chunk in the vector store."""
        return f"{self.node_id}_{self.chunk_index}"

    def to_metadata(self) -> dict:
        """Metadata dict for vector store ingestion."""
        return {
            "node_id": self.node_id,
            "node_name": self.node_name,
            "indexed_version": self.indexed_version,
            "chunk_index": self.chunk_index,
            "workspace": self.workspace_name,
            "doc_class": self.doc_class,
            "status": self.status,
            "modify_date": self.modify_date,
        }

At every pipeline stage boundary, the rule is: accept OTCSChunk, return OTCSChunk. Your text extractor creates it. Your chunker splits it while copying metadata to each split. Your embedder reads embed_text. Your vector store ingestor reads chunk_id(), embed_text, and to_metadata(). None of these stages sees or passes raw strings.

Building the Source Link

Once node_id is preserved through retrieval, building the clickable source link is straightforward. Your answer generation layer receives retrieved chunks with their metadata intact — use the node_id to construct the OTCS deep link for each cited source.

citations.py — build OTCS source links from retrieved chunk metadata
from dataclasses import dataclass
from urllib.parse import quote

@dataclass
class Citation:
    node_id: int
    node_name: str
    url: str
    version: int
    status: str

def build_otcs_link(
    node_id: int,
    base_url: str,
    ui: str = "smart"  # "smart" | "classic"
) -> str:
    """Build a direct-open link for an OTCS node."""
    base = base_url.rstrip("/")
    if ui == "smart":
        return f"{base}/otcs/cs.exe/open/{node_id}"
    else:
        return (
            f"{base}/otcs/cs.exe"
            f"?func=ll&objType=1&objID={node_id}"
        )

def build_citations(
    retrieved_chunks: list[dict],
    otcs_base_url: str,
    deduplicate: bool = True
) -> list[Citation]:
    """Extract unique citations from retrieved chunk metadata."""
    seen: set[int] = set()
    citations: list[Citation] = []

    for chunk in retrieved_chunks:
        meta = chunk.get("metadata", {})
        node_id = meta.get("node_id")
        if not node_id:
            continue
        if deduplicate and node_id in seen:
            continue
        seen.add(node_id)
        citations.append(Citation(
            node_id=node_id,
            node_name=meta.get("node_name", f"Node {node_id}"),
            url=build_otcs_link(node_id, otcs_base_url),
            version=meta.get("indexed_version", 0),
            status=meta.get("status", ""),
        ))
    return citations

The citation list is passed to your UI layer, which renders each source as a named link: "Source: Annual EHS Report 2023 → open in OpenText." The user clicks and lands directly on the document. No search, no navigation, no guessing.

✓

Pass citations alongside the LLM answer, not through the LLM: Don't ask the LLM to generate clickable URLs. LLMs hallucinate URLs reliably and creatively. Build the citation list programmatically from the retrieved chunk metadata, and render it in your UI layer. Keep the LLM responsible for synthesising the answer; keep your code responsible for tracing where it came from.

The Versioning Edge Case

OTCS maintains full version history. When a user checks in a new version of a document, the DataID stays the same but the version number increments. Your index was built against version 4 of the safety policy. The user has since checked in version 5 with a significant amendment. Your RAG answer is based on the v4 content but links to the current document — which is now v5 and says something different.

You have two options for managing this, and you should implement both:

  • Store indexed_version in chunk metadata. At retrieval time, compare it to the document's current version (via GET /api/v2/nodes/{node_id}). If they differ, surface a staleness warning alongside the citation.
  • Link to the node (current version) but note the indexed version in the audit record. The user wants to see the current document. The audit trail needs to know which version was used to generate the answer. Store both.
staleness_check.py — detect updated documents at query time
import requests

def check_staleness(
    node_id: int,
    indexed_version: int,
    ticket: str,
    otcs_base_url: str
) -> dict:
    """
    Returns staleness status for a cited node.
    Adds ~30ms latency per unique citation — cache aggressively.
    """
    url = f"{otcs_base_url}/api/v2/nodes/{node_id}"
    resp = requests.get(url, headers={"OTCSTicket": ticket}, timeout=5)
    if resp.status_code != 200:
        return {"stale": False, "current_version": None}

    data = resp.json().get("data", {})
    current_version = data.get("version", indexed_version)
    is_stale = current_version > indexed_version

    return {
        "stale": is_stale,
        "current_version": current_version,
        "indexed_version": indexed_version,
        "warning": (
            f"Document updated (v{indexed_version} → v{current_version})"
            if is_stale else None
        )
    }

A staleness warning surfaced in the UI — "Note: this document has been updated since it was indexed. Current version is v5; this answer is based on v4." — is enormously valuable for compliance use cases. It tells the user that the AI's answer may not reflect the current state of the document, without breaking the citation or removing the source link.

The Audit Trail Benefit

When node_id is preserved end to end, your system can produce a complete audit record for every AI interaction: which user, which query, which chunks were retrieved, which nodes were cited. In enterprise, particularly in regulated industries, this isn't a nice-to-have — it's a sign-off requirement for AI deployment.

The audit record structure for a single query looks like:

  • Timestamp and user identity (from the OTCSTicket or your auth layer)
  • Query text (sanitised — strip any sensitive terms before logging)
  • Retrieved node IDs and their chunk indices
  • Final cited node IDs (deduplicated) with version numbers
  • LLM model and version used for generation

This record correlates directly with the OTCS audit log. OTCS records every document access — if your RAG pipeline retrieves content on behalf of a user, that access appears in the OTCS audit tables via the OTCSTicket. Your RAG audit log, combined with the OTCS audit log, gives compliance teams a complete picture: not just "user accessed document" but "user asked this question, AI retrieved these documents, AI cited these nodes in the answer."

ℹ

OTCS audit events are accessible via GET /api/v2/nodes/{node_id}/audit and through the Audit Management module if your deployment has it licensed. For real-time audit event streaming, the OTCS Event Framework (where available) emits document access events that can be consumed by an external audit system.

5 Lines of Code. Months of Trust.

The actual code change to add node_id preservation to a naive pipeline is small. You're adding a field to your chunk dataclass, ensuring it's populated at extraction time, passing the dataclass through chunking instead of raw strings, storing the field in vector store metadata, and returning it from your retrieval function. If your pipeline is well-structured, this is a one-afternoon change.

The cost of not doing it: every citation your system generates is a dead end. Users who care about sources — and in enterprise, the users who matter most always care about sources — will learn quickly that the AI's citations can't be followed. They'll stop trusting the answers, even the correct ones. The RAG system becomes a novelty rather than a workflow tool.

RAG without source traceability is not enterprise-grade. It's a demo. Node ID preservation is the minimum technical requirement for an enterprise AI system to earn and maintain user trust. Build it in from day one, because adding it later means reindexing your corpus, and that's a conversation nobody wants to have six months into a production deployment.
