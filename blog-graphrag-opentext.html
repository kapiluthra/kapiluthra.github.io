GraphRAG on OpenText: Building a Knowledge Graph from Your Content Repository

Flat vector search returns the most similar documents. But enterprise questions are rarely flat — they span multiple documents, authors, workspaces, and relationships. GraphRAG uses the structure already in your ECM to answer them.

Kapil Uthra June 2026 ~14 min read
graphrag rag opentext knowledge graph architecture

When Microsoft released GraphRAG as open-source in 2024, the AI community noticed. The results on complex, multi-hop questions were dramatically better than flat vector search — in some evaluations, 3–4× more accurate on queries that required synthesising information across multiple documents. The reason: graphs capture relationships, and enterprise content is fundamentally relational.

What the community didn't focus on is that enterprise content management systems like OpenText Content Server are already graph structures. Every folder is a node. Every document is a node. Categories, workspaces, users, workflows — all nodes. The relationships between them — contains, authored-by, classified-as, approved-by — are all edges. You don't need to build a knowledge graph from scratch. You need to expose the one that already exists.

Where Flat RAG Hits Its Ceiling

Flat vector search works well for a specific class of questions: "find documents that talk about X." It fails — or produces hallucinated answers — on a different class: questions that require traversing relationships across documents.

Here are real examples I've encountered in enterprise ECM deployments:

  • "Find all contracts referencing Supplier ABC that were reviewed by the legal team after the 2023 policy change."
  • "Which projects have open change requests that were created by engineers who joined after the system upgrade?"
  • "Show me all documents in the Quality workspace that are linked to this batch record and haven't been signed off."

These are multi-hop queries. Each one requires traversing at least three entity types with specific relationship constraints. A vector store returns a flat list of similar chunks — it has no concept of "was reviewed by" or "is linked to." GraphRAG does.

OpenText as a Graph: The Entities and Relationships

Before building anything, map out what already exists in your OTCS instance as a graph structure. The node types are well-defined:

Node Types

  • Document — the content node (node_type: 144). Has version history, metadata categories, MIME type.
  • Folder / Workspace — container nodes (node_type: 0, 848). The structural scaffold of the repository.
  • Category — classification nodes. Documents can have multiple categories with attribute sets.
  • User — identity nodes. Author, modifier, assignee.
  • Workflow — process nodes. Maps to documents that have been through approval chains.

Edge Types (Relationships)

  • CONTAINS — Folder → Document, Workspace → Folder
  • AUTHORED_BY — Document → User (create_user_id)
  • MODIFIED_BY — Document → User (modify_user_id)
  • CLASSIFIED_AS — Document → Category
  • APPROVED_BY — Document → User (via workflow audit trail)
  • REFERENCES — Document → Document (via compound documents, URLs in content, or explicit relationship objects)
ℹ

OTCS also has a native Relationships feature (node_type: 1) that creates explicit typed links between nodes. If your implementation uses these, they're the highest-value edges in your graph — they represent intentional human-defined relationships, not just structural containment.

Extracting the Graph from OTCS

The graph extractor pulls the structural graph from OTCS via its REST API. You don't need to read document content at this stage — you're extracting metadata and relationships first.

graph_extractor.py — extract nodes and edges from OTCS
import requests
from dataclasses import dataclass
from typing import Iterator

@dataclass
class GraphNode:
    id: int
    type: str          # "document" | "folder" | "user" | "category"
    label: str
    properties: dict

@dataclass
class GraphEdge:
    source_id: int
    target_id: int
    relationship: str  # "CONTAINS" | "AUTHORED_BY" | "CLASSIFIED_AS" etc.
    properties: dict

class OTCSGraphExtractor:
    def __init__(self, base_url: str, ticket: str):
        self.base_url = base_url
        self.headers = {"OTCSTicket": ticket}

    def extract_workspace(self, root_id: int) -> tuple[list[GraphNode], list[GraphEdge]]:
        nodes, edges = [], []
        for item in self._walk(root_id):
            node = GraphNode(
                id=item["id"],
                type="document" if item["type"] == 144 else "folder",
                label=item["name"],
                properties={
                    "create_date": item.get("create_date"),
                    "modify_date": item.get("modify_date"),
                    "mime_type": item.get("mime_type"),
                }
            )
            nodes.append(node)

            # CONTAINS edge: parent → this node
            edges.append(GraphEdge(
                source_id=item["parent_id"],
                target_id=item["id"],
                relationship="CONTAINS",
                properties={}
            ))

            # AUTHORED_BY edge
            if item.get("create_user_id"):
                edges.append(GraphEdge(
                    source_id=item["id"],
                    target_id=item["create_user_id"],
                    relationship="AUTHORED_BY",
                    properties={}
                ))

        return nodes, edges

    def _walk(self, node_id: int) -> Iterator[dict]:
        url = f"{self.base_url}/api/v1/nodes/{node_id}/nodes"
        page = 1
        while True:
            resp = requests.get(url, headers=self.headers,
                               params={"page": page, "limit": 100})
            data = resp.json()
            for item in data.get("data", []):
                yield item
                if item["type"] == 0:  # recurse into folders
                    yield from self._walk(item["id"])
            if not data.get("collection", {}).get("paging", {}).get("next"):
                break
            page += 1

Enriching with LLM Entity Extraction

The structural graph gives you containment and authorship relationships. To answer business queries, you need a second layer: entities extracted from document content. This is where the LLM comes in.

For each document, you run an entity extraction prompt that identifies and returns:

  • Named entities: companies, people, locations, products
  • Key concepts: regulatory references, process names, system names
  • Relationships: "Document X references Supplier Y" or "Process A depends on System B"
Entity extraction prompt template
"""
Extract entities and relationships from this enterprise document.
Return a JSON object with:
  - entities: list of {name, type, description}
  - relationships: list of {source, relation, target}

Entity types: COMPANY, PERSON, SYSTEM, PROCESS, REGULATION, PRODUCT
Relationship types: REFERENCES, REQUIRES, APPROVES, SUPERSEDES, APPLIES_TO

Document text:
{document_text}

Respond with valid JSON only. No explanation.
"""

These extracted entities and relationships become additional nodes and edges in your graph, enriching it beyond what the OTCS metadata alone provides. A procurement contract now has edges to the supplier entity, the regulatory framework it references, and the systems it impacts — none of which appear in OTCS metadata but are present in the document body.

Query Patterns: What GraphRAG Unlocks

With the combined structural + semantic graph in place, you can answer query types that were previously impossible with flat RAG.

Multi-hop Entity Queries

User asks: "Which contracts reference SABIC and are located in the Procurement workspace?"

Graph traversal: COMPANY(SABIC) ← REFERENCES ← DOCUMENT → CONTAINS → WORKSPACE(Procurement). This is a two-hop query that finds the intersection — impossible with a single vector search, trivial with a graph.

Relationship Chain Queries

User asks: "Show me all documents approved by the legal team that also appear in the active projects list."

Graph traversal: USER(Legal Group) ← APPROVED_BY ← DOCUMENT → CONTAINS → WORKSPACE(Active Projects). Two separate chains, joined on Document.

Impact Analysis

User asks: "Which documents would be affected if we update the ISO 9001 quality procedure?"

Graph traversal: start from REGULATION(ISO 9001), follow APPLIES_TO edges to find all referencing documents, then expand to their workspaces and related processes.

⚠

GraphRAG is expensive at query time. Each multi-hop traversal may require multiple LLM calls to decompose the query into graph steps, execute traversals, and synthesise the answer. For simple lookup queries, flat RAG is faster and cheaper. Reserve GraphRAG for the complex relational queries where it genuinely outperforms.

The Query Classifier is a lightweight LLM call (or a rules-based classifier) that determines: does this query require relationship traversal (graph path) or content similarity search (vector path)? Many queries benefit from both — retrieve related documents via graph, then rank them by semantic similarity to the query.

When to Use GraphRAG vs Plain Hybrid Search

GraphRAG adds real operational cost: extraction pipeline, graph database to maintain, more complex query logic, higher latency. It's not always worth it. Use it when your users ask relationship queries that flat search can't answer. Don't use it if your corpus is small, queries are simple, or you need sub-second response times.

The honest benchmark from my testing: on a 500-document enterprise corpus, GraphRAG outperformed flat hybrid search on roughly 30% of queries — specifically those that spanned multiple document relationships. On the other 70%, the difference was negligible. That 30% subset was, however, the highest-value queries the business actually cared about.

✓

Start with the structural graph only (containment, authorship, categories). This is cheap to extract, requires no LLM at extraction time, and already unlocks workspace and author-based multi-hop queries. Add LLM entity extraction as a second phase once you've validated the use case with real users.

OpenText Content Server isn't just a document store. It's a structured knowledge graph that most organisations have never thought to query as one. GraphRAG is the mechanism that finally lets you ask it the questions it was built to answer.
