RAG Evaluation with RAGAS: Measuring What Actually Matters on an Enterprise OTCS Corpus

You can't improve a RAG pipeline you can't measure. Here's how to build a RAGAS evaluation harness against an OpenText Content Server corpus — generating ground truth, running metrics, and interpreting the results that matter.

Kapil Uthra May 2026 ~15 min read
ragragasevaluationopentextpython

Most RAG pipeline articles end at "the chatbot answers questions." The harder, less glamorous question is: how do you know if the answers are good? And more specifically — how do you know when a change to your chunking strategy, your embedding model, or your retrieval parameters actually improved things rather than just shifting the failure modes around?

The answer is systematic evaluation. RAGAS (Retrieval Augmented Generation Assessment) is currently the most practical framework for this, and it maps cleanly onto the specific challenges of an enterprise ECM corpus: documents with complex structure, domain-specific terminology, and ACL-sensitive content.

This post walks through building a full evaluation harness — from generating a ground truth dataset against your OTCS corpus, through running RAGAS metrics, to interpreting results and using them to make concrete pipeline improvements. The companion repo is at github.com/kapiluthra/otcs-ragas-eval.

The Three Metrics That Actually Matter

RAGAS exposes a lot of metrics. For an enterprise document retrieval system, three are load-bearing:

MetricWhat it measuresWhy it matters for ECM
faithfulness Are claims in the answer grounded in retrieved context? Enterprise users act on answers. A hallucinated contract term or compliance date is a real business risk — not just an LLM curiosity.
answer_relevancy Does the answer actually address the question asked? ECM queries are often specific ("what is the notice period in vendor contract X?"). A vague answer about contracts in general scores low.
context_recall Did retrieval surface the right documents for the question? The most actionable metric — if this is low, the problem is in your retrieval (chunking, embedding, hybrid search) not your LLM prompt.

A fourth metric, context_precision, measures whether retrieved chunks are relevant rather than noisy — useful once faithfulness and recall are in good shape.

Generating a Ground Truth Dataset

RAGAS requires a dataset of (question, ground_truth_answer, context) triples. For a corporate ECM corpus you can't use public benchmarks — you need domain-specific questions about your actual documents. There are two approaches:

  • Manual curation: ask 5-10 people who use the documents daily to write 5 questions each with answers they know. Slow but high quality.
  • LLM-generated with human review: use an LLM to generate question/answer pairs from document chunks, then have a domain expert review and correct them. Faster and scales to larger corpora.

I use the second approach with a light human review pass. The key is generating questions that reflect how real users actually query the system — not "what is this document about" but "what are the termination conditions in section 4.2".

generate_eval_dataset.py — LLM-generated Q&A pairs from OTCS chunks
import json
import requests
import pandas as pd
from pathlib import Path

OLLAMA_BASE = "http://localhost:11434"
CHAT_MODEL = "llama3"
PARQUET_PATH = "./cold_store.parquet"

GENERATION_PROMPT = """You are generating evaluation data for a RAG system.
Given the following document excerpt, generate ONE specific, 
answerable question that a business user might ask, and the 
correct answer based ONLY on the excerpt.

Rules:
- Question must be answerable from the excerpt alone
- Answer must be specific (not vague)  
- Prefer questions about facts, dates, names, conditions, or procedures
- Do NOT ask "what is this document about?"

Excerpt:
{text}

Respond as JSON only: {{"question": "...", "answer": "..."}}"""

def generate_qa_pair(text: str) -> dict | None:
    resp = requests.post(
        f"{OLLAMA_BASE}/api/chat",
        json={
            "model": CHAT_MODEL,
            "messages": [{"role": "user", 
                           "content": GENERATION_PROMPT.format(text=text[:1200])}],
            "stream": False,
        },
        timeout=60
    )
    content = resp.json()["message"]["content"]
    try:
        # Strip markdown fences if present
        clean = content.strip().strip("```json").strip("```").strip()
        return json.loads(clean)
    except json.JSONDecodeError:
        return None

def build_eval_dataset(n_samples: int = 100) -> pd.DataFrame:
    df = pd.read_parquet(PARQUET_PATH)
    # Sample from parent chunks (more context than child chunks)
    parents = df.drop_duplicates("parent_id").sample(
        min(n_samples * 2, len(df)), random_state=42
    )
    
    rows = []
    for _, row in parents.iterrows():
        qa = generate_qa_pair(row["parent_text"])
        if qa and qa.get("question") and qa.get("answer"):
            rows.append({
                "question": qa["question"],
                "ground_truth": qa["answer"],
                "context_source": row["doc_name"],
                "source_text": row["parent_text"],
            })
        if len(rows) >= n_samples:
            break
    
    result = pd.DataFrame(rows)
    result.to_csv("eval_dataset.csv", index=False)
    print(f"Generated {len(result)} Q&A pairs → eval_dataset.csv")
    return result
⚠

Review before trusting. LLM-generated ground truth is a starting point, not gospel. Have a domain expert review at least 20% of the generated pairs before running RAGAS — low-quality ground truth produces meaningless metrics. I flag generated answers containing hedging language ("may", "might", "could be") for mandatory human review.

Running the Pipeline and Collecting Answers

For each question in the eval dataset, you run your RAG pipeline and collect the answer along with the retrieved context chunks. These form the inputs to RAGAS.

run_eval_pipeline.py — Collect answers and contexts
import sys, os
sys.path.insert(0, "../otcs-rag-demo/src")
from retriever import HybridRetriever
import requests, json, pandas as pd

OLLAMA_BASE = os.getenv("OLLAMA_BASE_URL", "http://localhost:11434")
CHAT_MODEL = os.getenv("CHAT_MODEL", "llama3")

def answer_question(question: str, retriever: HybridRetriever) -> dict:
    results = retriever.search(question, top_k=4)
    contexts = [r["parent_text"] for r in results]
    context_str = "\n\n---\n\n".join(
        f"[{i+1}] {r['doc_name']}\n{r['parent_text']}" 
        for i, r in enumerate(results)
    )
    resp = requests.post(
        f"{OLLAMA_BASE}/api/chat",
        json={"model": CHAT_MODEL, "stream": False,
              "messages": [
                  {"role": "system", "content": "Answer using ONLY the provided context."},
                  {"role": "user", "content": f"Context:\n{context_str}\n\nQuestion: {question}"}
              ]},
        timeout=120
    )
    answer = resp.json()["message"]["content"]
    return {"answer": answer, "contexts": contexts}

def run_pipeline_on_dataset(eval_df: pd.DataFrame) -> pd.DataFrame:
    retriever = HybridRetriever()
    results = []
    for _, row in eval_df.iterrows():
        out = answer_question(row["question"], retriever)
        results.append({
            "question": row["question"],
            "answer": out["answer"],
            "contexts": out["contexts"],
            "ground_truth": row["ground_truth"],
        })
    return pd.DataFrame(results)

Running RAGAS Metrics

evaluate.py — Run RAGAS metrics
from datasets import Dataset
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_recall,
    context_precision,
)
from langchain_community.chat_models import ChatOllama
from langchain_community.embeddings import OllamaEmbeddings
import pandas as pd

def run_ragas_eval(results_df: pd.DataFrame) -> dict:
    # Use local Ollama for evaluation (no OpenAI needed)
    llm = ChatOllama(model="llama3", base_url="http://localhost:11434")
    embeddings = OllamaEmbeddings(
        model="nomic-embed-text", 
        base_url="http://localhost:11434"
    )
    
    dataset = Dataset.from_dict({
        "question": results_df["question"].tolist(),
        "answer": results_df["answer"].tolist(),
        "contexts": results_df["contexts"].tolist(),
        "ground_truth": results_df["ground_truth"].tolist(),
    })
    
    scores = evaluate(
        dataset,
        metrics=[faithfulness, answer_relevancy, context_recall, context_precision],
        llm=llm,
        embeddings=embeddings,
    )
    
    return scores.to_pandas()

What the Numbers Actually Mean

Here are the baseline scores from running this against a 500-document OTCS corpus with the hybrid BM25+vector pipeline described in the main RAG pipeline post:

MetricScoreInterpretation
faithfulness
0.71
71% of answer claims are grounded in retrieved context. The 29% gap is mostly the LLM adding general knowledge beyond what the documents say — addressable with a stricter system prompt.
answer_relevancy
0.79
79% of answers are actually addressing the question. The failures are mostly over-verbose answers that bury the specific answer in general context.
context_recall
0.68
Only 68% of questions have the relevant document in the top-4 retrieved chunks. This is the biggest opportunity — retrieval quality is the bottleneck, not the LLM.
context_precision
0.74
74% of retrieved chunks are relevant to the question. The noisy 26% mostly comes from broad queries that match too many documents.

Using Scores to Make Pipeline Decisions

The evaluation loop is only valuable if you use it to make concrete changes. Here's how each low score maps to a specific fix:

  • Low faithfulness (below 0.70): tighten the system prompt — add explicit instructions like "only state facts that appear verbatim in the provided context" and "if the context doesn't contain the answer, say so explicitly". Re-run eval. If it doesn't improve, the LLM is too small for this task.
  • Low answer relevancy (below 0.75): check if the retriever is returning the right chunks but the answer is burying the key fact. A prompt change asking for concise, direct answers often fixes this. Also check whether the questions in your eval dataset are too vague.
  • Low context recall (below 0.70): this is a retrieval problem. Experiment with: increasing top_k from 4 to 6, adjusting the BM25/vector RRF blend, trying a different chunk size, or improving document coverage in the corpus.
  • Low context precision (below 0.65): your retriever is surfacing noise. Try stricter metadata filtering, reduce top_k, or add a cross-encoder reranking step after initial retrieval.
✓

Run eval before and after every significant change. Treat it like a test suite. Changing chunk size from 400 to 600 tokens improved context_recall from 0.68 to 0.74 on my corpus — a 9% improvement that I would have missed without measurement.

The Enterprise-Specific Wrinkles

A few things that make RAGAS evaluation harder in an enterprise ECM context compared to general RAG tutorials:

  • Domain terminology: RAGAS uses embedding similarity to score answer_relevancy. Generic embeddings may not understand that "OTCSTicket" and "authentication token" are the same concept in your domain. Consider fine-tuning or using domain-adapted embeddings for evaluation too.
  • ACL-filtered results: if your retriever enforces ACL gates, some questions will fail to retrieve any context (the relevant document is inaccessible to the evaluation user). These questions should be excluded from evaluation rather than counted as retrieval failures — otherwise your context_recall metric is artificially depressed.
  • Document structure: enterprise documents often have headers, tables, and numbered clauses that chunk badly. If your eval dataset has many questions about tabular data and your chunker destroys table structure, context_recall will be low for structural reasons rather than retrieval reasons. Segment your eval dataset by document type to diagnose this.
